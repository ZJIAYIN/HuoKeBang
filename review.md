## 1. 项目概览

EchoMind 是一个基于 **Claude API (Anthropic)** 的企业级智能客服系统。它采用 **多 Agent 编排架构**，将复杂客服场景分解为：意图识别 → Agent 路由 → 工具调用 → 记忆管理 → 监控反馈 的完整链路。

| 维度 | 说明 |
|------|------|
| **技术栈** | Python 3.12 + FastAPI + Anthropic SDK + Redis + ChromaDB + Prometheus |
| **部署方式** | Docker Compose 多服务编排（5 个服务） |
| **AI 能力** | 仅依赖 Anthropic Claude API，无本地模型 |
| **向量数据库** | ChromaDB（内置 all-MiniLM-L6-v2 Embedding） |
| **代码规模** | ~1500 行核心 Python 代码，7 个模块文件 |

## 3. 模块深度分析

### 3.1 API 接入层 — `api/main.py`

**定位**：系统入口，FastAPI 应用，负责 HTTP 路由、组件初始化、请求/响应编解码。

**核心职责**：

| 路由 | 功能 | 说明 |
|------|------|------|
| `GET /health` | 健康检查 | 返回 Agent 统计快照 |
| `POST /chat` | **主对话接口** | 完整的 5 步流程 |
| `GET /monitor` | 监控摘要 | Agent/工具统计 + 告警 + 建议 |
| `GET /metrics` | Prometheus 指标 | Prometheus 拉取 |
| `POST /search` | RAG 检索演示 | 展示查询改写 + 重排链路 |
| `POST /knowledge/add` | 批量导入知识库文档 | JSON 格式 |
| `POST /knowledge/upload` | 文件上传导入 | txt/md/json |
| `GET /knowledge/stats` | 知识库统计 | 文档片段总数 |
| `POST /eval/run` | 运行评测 | 内置/自定义用例 |

**关键设计**：

- **lifespan 生命周期管理**：所有全局组件（Orchestrator、Memory、ToolManager、Monitor、Evaluator）在 lifespan 中初始化，确保服务就绪后才接受请求
- **RAG 主动跳过策略**：`_should_use_knowledge()` 对纯寒暄（问候语列表）跳过检索，业务关键词触发检索，既节省 API 调用又避免干扰日常对话
- **非阻塞画像更新**：`asyncio.create_task(_memory.update_profile(...))` 在响应返回后异步更新用户画像，不增加主链路延迟

### 3.2 Agent 编排器 — `agents/agent_orchestrator.py`

**定位**：系统的"大脑"，负责 Agent 路由、执行调度、降级和升级决策。

**三层路由策略**：

```
用户消息
    │
    ▼
┌─────────────────────────────┐
│  第 1 层：意图路由            │
│  IntentCategory → AgentType  │  ← 静态映射表
│  TECHNICAL→TechnicalAgent    │
│  BILLING→BillingAgent        │
└─────────────┬───────────────┘
              │
              ▼
┌─────────────────────────────┐
│  第 2 层：性能路由            │
│  routing_score() 选最优 Agent│  ← 动态（Monitor 反馈）
│  成功率 × 70% + 延迟 × 30%   │
│  再乘以 (1 - monitor_penalty) │
└─────────────┬───────────────┘
              │
              ▼
┌─────────────────────────────┐
│  第 3 层：降级路由            │
│  专属 Agent 失败 → General   │  ← 兜底策略
│  Agent 池为空 → 错误信息      │
└─────────────┬───────────────┘
              │
              ▼
          Agent.handle()
```

**并行协作机制**：

`_collaboration_targets()` 通过领域关键词检测复合问题（如"登录报错且被重复扣款"），将请求同时派发给 Technical + Billing Agent，结果异步合并返回。这是系统处理复杂问题的重要能力。

**关键设计亮点**：

- **回合制统计收敛**：`AgentStats.routing_score()` 是闭环关键——成功率降低 → routing_score 下降 → 路由概率降低 → 该 Agent 压力减轻 → 有机会恢复
- **监控反馈接口**：`update_routing_penalties()` 接收 Monitor 计算的路由惩罚值，实现"在线表现 → 路由权重调整"的负反馈闭环
- **升级检测**：`_needs_escalation()` 关键词检测 + CRITICAL 紧急度强制升级，双重保障

### 3.3 意图识别器 — `core/intent_recognizer.py`

**定位**：系统的"耳朵"，理解用户意图，为 Agent 路由提供依据。

**三路融合策略**：

```
用户消息
    │
    ├──────────────────────► LLM 语义理解 (权重 70%)
    │                            │
    │                    Few-shot 示例 + 对话历史上下文
    │                            │
    ├──────────────────────► Embedding 向量相似度 (权重 20%)
    │                            │
    │                    n-gram 哈希向量 vs 模板 Embedding
    │                    余弦相似度取最高分
    │                            │
    └──────────────────────► 关键词模式匹配 (权重 10%)
                                 │
                          零延迟兜底，硬关键词命中
                                 │
                                 ▼
                        ┌─────────────────┐
                        │   加权投票合并    │
                        │  阈值过滤 (0.5)  │
                        │     → 输出意图    │
                        └─────────────────┘
```

**关键的 Embedding 降级设计**：

```python
async def _embed_text(self, text: str) -> List[float]:
    # 优先远端 Embedding API
    if has_embeddings_client:
        return await remote_embedding(text)
    # 远端不可用时使用本地 n-gram 哈希向量
    return self._local_embedding(text)  # 纯 Python，不依赖任何第三方
```

这个设计保障了"三路融合"结构在任何部署环境下都不会因为 Embedding 缺失而断裂。

**其他设计亮点**：

- **缓存策略**：LRU 缓存（上限 1000 条），通常场景文本去重后命中率可观
- **在线学习**：`learn()` 可将新样本加入模板库，实现持续改进
- **实体提取**：LLM 从消息中抽取结构化实体（订单号、金额、错误码等），为后续 Agent 提供结构化上下文
- **紧急度评估**：关键词 + 意图双维度，CRITICAL → 直接触发升级流程

### 3.4 MCP 工具框架与 RAG 知识库 — `mcp/` 目录

#### 3.4.1 工具管理器 — `mcp/tool_manager.py`

**定位**：系统的"手"，封装工具调用的可靠性机制。

**核心执行链**：

```
用户调用工具
    │
    ▼
┌────────────────┐
│ 1. 缓存检查     │ ← TTL 缓存，相同参数直接返回
└───────┬────────┘
        │ 未命中
        ▼
┌────────────────┐
│ 2. 熔断检查     │ ← 三态熔断器（CLOSED→OPEN→HALF_OPEN）
└───────┬────────┘
        │ 允许
        ▼
┌────────────────┐
│ 3. 参数校验     │ ← JSON Schema 校验 required + type
└───────┬────────┘
        │ 通过
        ▼
┌────────────────┐
│ 4. 执行 + 超时  │ ← asyncio.wait_for(timeout_s)
└───────┬────────┘
        │ 成功/失败
        ▼
┌────────────────┐
│ 5. 缓存写入     │
└───────┬────────┘
        │
        ▼
┌────────────────┐
│ 6. 可选重排     │ ← LLM 对结果列表按相关性重排序
└───────┬────────┘
        │
        ▼
    ToolResult
```

**检索优化链路**（`search_with_rewrite` — 解决"召回不全、排序差"的核心方案）：

```
用户查询 "退款流程"
    │
    ▼
┌────────────────┐
│ 查询改写 (LLM)  │ → ["退款流程", "如何申请退款", "退款需要多少天", "退款政策是什么"]
│ n=3 多角度扩写   │
└───────┬────────┘
        │
        ▼
┌────────────────┐
│ 并行召回         │ → 所有子查询同时检索 ChromaDB
│ asyncio.gather  │
└───────┬────────┘
        │
        ▼
┌────────────────┐
│ 合并去重         │ → 按内容 MD5 去重，消除冗余
│ (seen set)      │
└───────┬────────┘
        │
        ▼
┌────────────────┐
│ LLM 重排        │ → LLM 对候选集打分排序
│ rerank → Top-K  │ → 向量相似度 ≠ "对用户有用"
└───────┬────────┘
        │
        ▼
   高质量 Top-K 结果
```

**熔断器状态机**：

```
CLOSED ──(连续失败 ≥ threshold, 默认 5 次)──→ OPEN
  ↑                                            │
  │                (recovery_s, 默认 60s)       │
  │                                            ▼
  └────────────── HALF_OPEN ──(成功 1 次)──────┘
                      │
                      └──(再失败)──→ OPEN
```

#### 3.4.2 RAG 知识库 — `mcp/knowledge_base.py`

**定位**：系统的"长期记忆源"，基于 ChromaDB 的语义检索知识库。

**关键能力**：

- **文档切片**：按句子切分，每片 500 字，保留语义完整性
- **混合连接**：优先连接独立 ChromaDB 服务，不可用时降级为本地 PersistentClient
- **自动向量化**：ChromaDB 内置 all-MiniLM-L6-v2 模型，无需手动调用 Embedding API
- **默认知识库**：6 篇内置客服常见文档（退款、订单、账户、技术、会员、配送）

### 3.5 记忆管理器 — `memory/conversation_memory.py`

**定位**：系统的"记忆"，三级架构模拟人类记忆机制。

**三级记忆架构**：

```
┌────────────────────────────────────────────────────────────────┐
│                        三级记忆架构                              │
│                                                                │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  ① 工作记忆 (Redis)                                     │   │
│  │  存储：最近 20 条消息，TTL 24h                           │   │
│  │  特点：毫秒级读写，支持自动压缩（满 15 条触发）             │   │
│  │  操作：lpush + lrange, 最新在前                          │   │
│  │  压缩：LLM 摘要旧消息 → 存入情景记忆 → 保留最近 5 条      │   │
│  └─────────────────────────────────────────────────────────┘   │
│                               │                                 │
│                               ▼                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  ② 情景记忆 (ChromaDB collection "episodic")            │   │
│  │  存储：压缩后的历史对话片段，语义检索                      │   │
│  │  特点：跨会话检索，按语义相似度匹配                        │   │
│  │  查询：query_texts + where(user_id)                     │   │
│  └─────────────────────────────────────────────────────────┘   │
│                               │                                 │
│                               ▼                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  ③ 用户画像 (ChromaDB collection "user_profile")        │   │
│  │  存储：LLM 提炼的用户偏好 + 关键实体                      │   │
│  │  更新：异步，不阻塞主链路                                  │   │
│  │  覆盖：每次更新覆盖上次（同 conv_id 删除旧 doc 再写新）    │   │
│  └─────────────────────────────────────────────────────────┘   │
└────────────────────────────────────────────────────────────────┘
```

**压缩前/后对比**：

| 状态 | 存储位置 | 内容 | 大小 |
|------|---------|------|------|
| 压缩前 | Redis 列表 | 原始消息（USER + ASSISTANT 交替） | 最多 15 条 |
| 压缩后 | Redis 摘要 key | LLM 生成的 2-3 句摘要 + 历史摘要拼接 | 简短 |
| 压缩后 | Redis 列表 | 最近 5 条原始消息 | 5 条 |
| 压缩后 | ChromaDB episodic | 旧消息原文 + LLM 摘要 | 持久化 |

**上下文融合** (`to_prompt_text`)：

```
[会话摘要]         ← 从 Redis summary key 读取
最新对话摘要...

[相关历史]         ← 从 ChromaDB episodic 语义检索
- 上次也问过退款问题...

[用户画像]         ← 从 ChromaDB user_profile 读取
{"preferences": ["快速处理"], "entities": {...}}

[最近对话]         ← 最近 N 条原始消息
user: 我的订单还没到
assistant: 请问订单号是？

```

### 3.6 性能监控器 — `monitor/performance_monitor.py`

**定位**：系统的"神经系统"，实时感知 Agent 和工具的健康状态，反馈给路由层。

**监控闭环**：

```
Agent/Tool 处理请求
    │
    ▼  ← 实时更新 stats（成功/失败/延迟）
┌────────────────────┐
│  Monitor 采集循环   │  ← 每 N 秒执行一次（默认 10s）
│  _collect()        │
└────┬───────┬───────┘
     │       │
     ▼       ▼
┌────────┐ ┌──────────────┐
│异常检测 │ │ 阈值告警       │
│Z-score  │ │ 硬阈值判断     │
│窗口 60  │ │ webhook 通知  │
└────┬───┘ └──────┬───────┘
     │            │
     ▼            ▼
┌──────────────────────────────────────────────┐
│  routing_penalty()                            │
│  将在线表现转为 0-0.9 的路由降权系数             │
│  成功率 < 90% → +惩罚                          │
│  延迟 > 3000ms → +惩罚                         │
└───────────────────┬──────────────────────────┘
                    │
                    ▼
┌──────────────────────────────────────────────┐
│  Orchestrator.update_routing_penalties()      │
│  → Agent.routing_score() 降低                 │
│  → _best_agent() 自动绕开问题 Agent          │
└──────────────────────────────────────────────┘
```

**告警阈值**：

| 指标 | 阈值 | 严重级别 | 方向 |
|------|------|---------|------|
| Agent 成功率 | < 90% | ERROR | less_than |
| 工具成功率 | < 95% | WARNING | less_than |
| Agent 平均延迟 | > 3000ms | WARNING | greater_than |
| 工具平均延迟 | > 5000ms | ERROR | greater_than |

**路由惩罚计算**（`_routing_penalty`）：

```python
penalty = 0.0
if success_rate < 0.90:
    penalty += min(0.5, (0.90 - success_rate) * 2)  # 成功率越低惩罚越大
if avg_ms > 3000:
    penalty += min(0.4, (avg_ms - 3000) / 10000)    # 延迟越高惩罚越大
return min(penalty, 0.9)  # 最大 0.9
```

### 3.7 评测框架 — `evaluation/evaluator.py`

**定位**：系统的"体检中心"，用 LLM-as-Judge 实现可规模化的端到端评测。

**评测流程**：

```
┌──────────────────────────────────────────────────────────────┐
│                        EndToEndEvaluator.run()                  │
│                                                                │
│  ┌─────────────────────┐  ┌───────────────────────────────┐   │
│  │ 意图识别评测         │  │ 对话质量评测                    │   │
│  │                     │  │                                │   │
│  │ IntentTestCase × N  │  │ DialogCase × N                │   │
│  │     │               │  │     │                          │   │
│  │     ▼               │  │     ▼                          │   │
│  │ recognizer.recog()  │  │ orchestrator.run() → 回复      │   │
│  │     │               │  │     │                          │   │
│  │     ▼               │  │     ▼                          │   │
│  │ Accuracy / Macro-F1 │  │ LLM Judge 四维评分             │   │
│  │ 按类别 Precision    │  │ 相关/准确/完整/有用            │   │
│  │ Recall / F1        │  │     │                          │   │
│  └─────────┬───────────┘  │     ▼                          │   │
│            │              │ 综合评分 ≥ 0.75 → pass          │   │
│            │              └───────────────┬───────────────┘   │
│            │                              │                   │
│            └──────────┬───────────────────┘                   │
│                       ▼                                       │
│  ┌─────────────────────────────────────────────────────┐     │
```

## 6. 亮点总结

### 架构层面

1. **三层路由 + 闭环反馈**：意图路由 → 性能路由 → 降级路由，配合 Monitor 的在线表现反馈，形成一个完整的负反馈闭环。这是系统最核心的架构设计。

2. **三级记忆模拟人类记忆**：Redis 工作记忆（即时）+ ChromaDB 情景记忆（跨会话）+ 用户画像（长期），自动压缩机制防止上下文爆炸，设计优雅且贴近业务需求。

3. **MCP 工具框架的鲁棒性设计**：熔断器（防雪崩）+ TTL 缓存（降低延迟）+ Query Rewrite（提高召回）+ LLM 重排（提高准确）+ 降级策略（容错），5 重机制保障工具调用的稳定性和质量。

4. **评测闭环**：LLM-as-Judge + 回归检测 + 优化建议，让系统可以持续迭代改进。

### 代码层面

5. **三路融合意图识别**：LLM（70%）+ Embedding（20%）+ 模式匹配（10%）加权投票，三路并行调用，兼顾准确性和速度。Embedding 策略设计了完善的降级链路。

6. **自动压缩机制**：Redis 列表条数超过 15 时自动触发 LLM 摘要 + ChromaDB 持久化，优雅解决长对话上下文膨胀问题。

7. **非阻塞操作**：用户画像更新使用 `asyncio.create_task` 异步执行，不阻塞主响应链路。

8. **完善的错误处理**：每个模块都有 try-except 兜底，异常不传播到上层。工具调用有熔断、超时、降级三层保护。

### 运维层面

9. **Docker Compose 编排**：5 个容器通过健康检查链串联，确保启动顺序正确。

10. **Prometheus + Nginx**：生产级监控和反向代理配置，限流、负载均衡、安全头、SSL 备选一应俱全。

11. **Docker 多阶段构建**：从 3.5GB → ~500MB，预下载 ONNX 模型避免运行时下载超时。

## 7. 改进建议

### 架构 / 功能建议

| # | 建议 | 优先级 | 说明 |
|---|------|--------|------|
| 1 | **增加 Agent 类型扩展性** | 高 | 当前 Agent 硬编码为 General/Technical/Billing，建议通过配置或注册机制实现插件式扩展，支持动态添加新 Agent 类型 |
| 2 | **意图路由映射表可配置化** | 中 | `_INTENT_ROUTING` 是静态字典，建议改为配置文件或通过 API 动态更新，无需改代码即可调整路由策略 |
| 3 | **增加重试机制** | 中 | LLM 调用失败时没有自动重试，对于瞬态网络错误，重试 1-2 次能显著提高成功率 |
| 4 | **限流提升到应用层** | 中 | Nginx 层做了 10r/s 限流，但应用层没有。容器重启或 Nginx 直连时，限流会失效 |
| 5 | **升级检测增强** | 中 | 当前仅靠关键词检测 "转人工" 等，建议用 LLM 语义判断用户情绪/不满程度，更精准触发升级 |
| 6 | **并发安全** | 低 | `AgentStats` 在多协程写场景下存在竞态，建议加锁或使用 atomic 操作。目前单进程 + asyncio 下风险较低，但需留意 |

### 代码质量建议

| # | 建议 | 文件 | 说明 |
|---|------|------|------|
| 7 | **类型注解不完整** | 多处 | `_routing_penalty` 等方法缺少类型注解，`_build_knowledge_context` 返回值类型为 `tuple` 而非命名元组 |
| 8 | **重复的 `_clean_text` 工具函数** | recognizer.py + tool_manager.py + evaluator.py + memory.py | 4 个文件有完全相同的方法，应提取为公共工具模块 |
| 9 | **评测基线路径硬编码** | docker-compose.yml | `EVAL_BASELINE_PATH` 在 compose 中被覆写为固定路径，不够灵活 |
| 10 | **`_profile.delete` 裸 except** | memory.py:183 | 删除文档时捕获了所有 Exception，建议缩小异常范围或至少区分 "文档不存在" 和 "服务异常" |
| 11 | **`try-except` 粒度过粗** | 多处 | 部分 try 块范围过大，如 `_compress` 中 LLM 调用和 Redis 操作在一个 try 块里，建议拆分成更细的异常处理 |
| 12 | **`_cache` 过期清理性能** | tool_manager.py:376 | LRU 清理采用 `list[:1250]` 删除操作，O(n) 复杂度的 `del` 操作在 BTree 实现中性能可接受，但仍建议考虑 `OrderedDict` 或 `cachetools` |

### 测试 / 可靠性建议

| # | 建议 | 优先级 | 说明 |
|---|------|--------|------|
| 13 | **缺少单元测试** | 高 | 所有模块均无单元测试，是项目最大的质量隐患。至少应为 IntentRecognizer（三路融合逻辑）、AgentOrchestrator（路由决策）、MemoryManager（压缩逻辑）编写测试 |
| 14 | **评测用例覆盖有限** | 中 | 8 条意图用例 + 5 条对话用例，建议补充边界用例（空消息、超长消息、多语言混杂、emoji 等） |
| 15 | **知识库倒灌测试** | 中 | /knowledge/add 和 /knowledge/upload 没有重复文档检测，同一内容导入多次会产生大量冗余切片 |
| 16 | **客户端重试行为测试** | 低 | 熔断器恢复后没有客户端的重试测试，建议编写集成测试验证熔断 → 恢复 → 自动切回的完整链路 |

### 运维建议

| # | 建议 | 优先级 | 说明 |
|---|------|--------|------|
| 17 | **日志中心化** | 中 | 当前日志分散在容器中，建议集成 EFK/ELK 或 Loki 进行日志聚合 |
| 18 | **配置版本管理** | 中 | 部分配置（如 CLIENT_MAX_BODY_SIZE）在 nginx.conf 中硬编码，缺少版本管理 |
| 19 | **备份策略** | 中 | ChromaDB 持久化目录和 Redis AOF 文件没有备份方案 |
| 20 | **Grafana 面板** | 低 | `.env.example` 中配置了 Grafana 凭据但 Docker Compose 中没有 Grafana 服务，建议补充 |
