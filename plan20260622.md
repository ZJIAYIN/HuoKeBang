# EchoMind 架构演进计划 (2026-06-22)

## 背景

当前架构：意图 → 情绪 → Router → 特定 Agent

```
用户输入
  │
  ▼
intent_recognizer → {intent, sentiment}
  │
  ▼
Router
  │
  ├── ConsultAgent
  ├── EscalationAgent
  ├── LeadCaptureAgent
  └── GreetingAgent
  │
  ▼
回复
```

**问题**：真实用户往往是多意图的，一个意图路由到一个 Agent 的方式无法处理"投诉+询价+留资"等复合场景。强行顺序执行多个 Agent 会带来回复风格不一致、状态同步困难、Token 成本增加等问题。

---

## 目标架构：三层 Skill 编排

```
用户输入
  │
  ▼
┌─────────────────────────────────────────────┐
│  Planner（LLM）                              │
│  职责：理解用户，输出结构化信息                │
│  输出：                                      │
│  ├─ primary_intent（主意图，全量）            │
│  ├─ sub_tasks（子任务列表，全量）              │
│  ├─ slot_ops（槽位变更，增量 Diff）           │
│  └─ emotion（情绪）                          │
└─────────────────┬───────────────────────────┘
                  │
        ┌─────────┴──────────┐
        │                    │
        ▼                    ▼
┌──────────────┐   ┌─────────────────┐
│ Slot Manager │   │ Skill           │
│ (程序)       │   │ Orchestrator    │
│              │   │ (程序)          │
│ session      │   │                 │
│ .update()    │   │ sub_tasks       │
│ .delete()    │   │ + session_slots │
│              │   │ → completed     │
│              │   │   / pending     │
└──────┬───────┘   └────────┬────────┘
       │                    │
       └────────┬───────────┘
                ▼
┌─────────────────────────────────────────────┐
│  Response Agent（LLM）                       │
│  输入：completed_tasks + pending_tasks       │
│        + emotion + session_slots            │
│  职责：生成自然回复，追问缺失信息              │
│  输出：回复文本                              │
└─────────────────────────────────────────────┘
```

### 三层职责边界

| 层 | 组件 | 类型 | 职责 |
|---|------|------|------|
| 理解层 | Planner | LLM | 从自然语言提取意图、子任务、槽位变更、情绪 |
| 编排层 | Slot Manager + Orchestrator | 纯代码 | 维护会话状态、检查槽位完备性、编排 Skill |
| 生成层 | Response Agent | LLM | 根据编排结果生成自然回复（含追问）|

设计原则：
- **LLM 负责理解与生成**（它擅长的）
- **程序负责状态与判断**（它擅长的）
- **职责不重叠，不互抢**

---

## 关键设计决策

### 1. Slot 管理：增量 Diff + 会话级 Slot Store

Planner **不输出完整槽位**，只输出本轮变更（Diff）：

```json
// Planner 输出
{
  "primary_intent": "PURCHASE",
  "sub_tasks": ["PRICE", "LEAD_CAPTURE"],
  "slot_ops": [
    {"op": "SET", "slot": "model", "value": "M8"},
    {"op": "SET", "slot": "budget", "value": "20万"}
  ],
  "emotion": "neutral"
}
```

Slot Manager 执行确定性合并：

```python
for op in slot_ops:
    if op.op == "SET":
        session_slots[op.slot] = op.value
    elif op.op == "DELETE":
        session_slots.pop(op.slot, None)
```

**为什么不用全量输出？**
- 多轮后 slot 可能膨胀到十几个字段
- LLM 漏写一个字段，你无法判断是"用户要删除"还是"LLM 忘写了"
- Diff 模式下 LLM 只需要关注"这轮变化了什么"，任务更简单

**DELETE 触发条件**：仅当用户明确取消时才触发（如"这个预算不考虑了"），普通对话无需 DELETE。

### 2. sub_tasks：全量输出

与 slot 不同，sub_tasks 每轮全量输出，原因是：
- sub_tasks 数量少（通常 1~4 个）
- 是 Orchestrator 的编排依据
- 漏掉一个等于丢掉用户诉求

```json
// Turn 1: "我想买M8"
{"sub_tasks": ["PRICE", "PRODUCT"]}

// Turn 2: "20万"
{"sub_tasks": ["PRICE"]}  // 价格咨询是当前焦点
```

### 3. Emotion：不进 Slot，直接影响 Orchestrator 决策

emotion 的流向：
- **不进 Slot Store**（不是业务槽位）
- **不进 Response Agent 的语气控制**（不交给 LLM）
- **放 Orchestrator 代码层**，作为一个决策信号

主要影响：**LeadCaptureSkill 的追问资格**

```python
class LeadCaptureSkill:
    required_slots = ["phone"]
    
    def can_execute(self, session_slots, emotion):
        """情绪差不追问留资"""
        if emotion in ("angry", "very_negative", "skeptical"):
            return False, "用户情绪不佳，不适合追问留资"
        return super().can_execute(session_slots)
```

### 4. 未知 sub_task 的处理

如果 Planner 输出了一个不存在的 sub_task（如 `REFUND`），Orchestrator **直接跳过**，不做任何处理。由已有的 Skill（如 `ComplaintSkill`）兜底覆盖。

## Skill 接口定义

```python
class BaseSkill:
    """所有 Skill 的基类"""
    
    name: str                    # skill 标识，与 sub_task 对应
    required_slots: list[str]    # 必需的槽位列表
    optional_slots: list[str]    # 可选的槽位列表
    
    def can_execute(self, session_slots: dict, emotion: str = None) -> tuple[bool, list[str]]:
        """
        检查是否可以执行。
        返回 (是否可执行, 缺失槽位列表)
        """
        missing = [s for s in self.required_slots 
                   if s not in session_slots or session_slots[s] is None]
        if missing:
            return False, missing
        return True, []
    
    async def execute(self, session_slots: dict, context: dict) -> str:
        """执行 Skill，返回结果片段"""
        raise NotImplementedError
```

| Skill | sub_task 标识 | 必需槽位 | 说明 |
|-------|-------------|---------|------|
| ProductSkill | PRODUCT | `model` | 车型查询 |
| PriceSkill | PRICE | `model` | 价格查询 |

- **Prompt 模板**：ConsultAgent 的 prompt 模板部分内容可复用为 Response Agent

## 面试问题预设

### Q: 你这个架构解决了什么问题？

> 之前是"一个意图路由到一个 Agent"的简单映射，但真实用户往往一句话里有多个诉求，比如"投诉+询价+留资"。多 Agent 路由在这种场景下要么只能选一个处理，要么顺序执行多个 Agent，带来回复风格不一致和 Token 浪费。新架构让 LLM 做规划（Plan），确定主意图和子任务，程序做编排（Orchestrate），一个通用 Agent 按规划执行多个 Skill，一次生成完整回复。

### Q: Slot 为什么用 Diff 而不是全量？

> 多轮后槽位可能累积十几个字段，如果让 LLM 每轮全量输出，它漏掉一个你无法判断是故意删除还是生成遗漏。Diff 模式让 LLM 只说"这轮变化了什么"，状态由程序确定性维护，职责分离更干净。

### Q: 情绪在架构中扮演什么角色？

> 情绪不作为路由信号，而是直接影响 Orchestrator 的决策——比如用户情绪差时不追问留资。这个判断放在代码层而不是 LLM 层，确保行为是确定性的。

### Q: 新增一个能力需要改多少代码？

> 只需新增一个 Skill 文件，声明 `required_slots`，实现 `execute()` 方法。Planner Prompt 里加一个样例，Orchestrator 和 Response Agent 完全不用改。
