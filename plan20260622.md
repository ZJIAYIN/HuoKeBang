# EchoMind 架构演进计划 (2026-06-22)

## 背景

当前架构：意图 → 情绪 → Router → 特定 Agent

```
用户输入
  │
  ▼
intent_recognizer → {intent, sentiment}
  │
  ▼
Router
  │
  ├── ConsultAgent
  ├── EscalationAgent
  ├── LeadCaptureAgent
  └── GreetingAgent
  │
  ▼
回复
```

**问题**：真实用户往往是多意图的，一个意图路由到一个 Agent 的方式无法处理"投诉+询价+留资"等复合场景。强行顺序执行多个 Agent 会带来回复风格不一致、状态同步困难、Token 成本增加等问题。

---


### 四层职责边界

| 层 | 组件 | 类型 | 职责 |
|---|------|------|------|
| 理解层 | Planner | LLM | 从自然语言提取意图、子任务、槽位变更、情绪 |
| 编排层 | Slot Manager + Orchestrator | 纯代码 | 维护会话状态、检查槽位、收集 Tool、合并 Instruction |
| 工具层 | RAG / CRM / ... | 纯代码 | 统一执行 Skill 所需的工具，结果共享 |
| 生成层 | Response Agent | LLM | 根据 Instruction + Context 生成自然回复 |

设计原则：
- **LLM 负责理解与生成**（它擅长的）
- **程序负责状态、编排与工具调用**（它擅长的）
- **职责不重叠，不互抢**
- **同一 Tool 只执行一次，结果多 Skill 共享**

## 组件详解

### 1. Planner（LLM）

接收 MemoryManager 拼接的上下文 + 用户当前输入，输出结构化信息：

```json
{
  "primary_intent": "PURCHASE",
  "sub_tasks": ["PRICE", "PRODUCT", "LEAD_CAPTURE"],
  "slot_ops": [
    {"op": "SET", "slot": "model", "value": "M8"},
    {"op": "SET", "slot": "budget", "value": "20万"}
  ],
  "emotion": "neutral"
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `primary_intent` | 全量 | 主意图，当前对话最核心的意图 |
| `sub_tasks` | **全量** | 所有待处理的子任务，每轮完整输出 |
| `slot_ops` | **增量 Diff** | 只输出本轮变化（SET / DELETE）|
| `emotion` | 单值 | 用户当前情绪 |

---

### 2. 编排层（纯代码）

#### 2.1 Slot Manager

**职责**：维护会话级 Slot Store，执行确定性合并。

```python
class SlotManager:
    def __init__(self):
        self.session_slots = {}  # Redis / 内存
    
    def apply(self, slot_ops: list[SlotOp]):
        for op in slot_ops:
            if op.op == "SET":
                self.session_slots[op.slot] = op.value
            elif op.op == "DELETE":
                self.session_slots.pop(op.slot, None)
```

**为什么用 Diff？** 多轮后 slot 可能膨胀到十几个字段，LLM 每轮全量输出时漏写一个字段无法判断是"用户要删除"还是"LLM 忘写"。Diff 模式让 LLM 只说"这轮变化了什么"。

**DELETE 触发条件：** 仅当用户明确取消时才触发（如"这个预算不考虑了"）。普通对话无需 DELETE。

#### 2.2 Orchestrator

**职责**：接收 Planner 输出 → 执行编排 → 收集 Tool → 准备 Response Agent 输入。

流程：

```
① 遍历 sub_tasks，匹配 SkillRegistry
   ── 匹配不到 → 跳过（如 REFUND 不存在就跳过）
   ── 匹配到   → 进入下一步

② 检查 required_slots
   ── 缺失 → 加入 pending
   ── 满足 → 进入下一步

③ 检查 emotion 条件
   ── 如 LeadCaptureSkill 在情绪差时不可执行 → 加入 pending

④ 满足条件的 Skill → 收集它们的 required_tools
   ── RAG: True/False
   ── CRM: True/False（未来）
   ── ...

⑤ 收集所有可执行 Skill 的 instruction
   ── 拼成 Response Agent 的 Instruction 部分

⑥ 统一执行 Tool Layer
   ── 需要 RAG → 执行一次 RAG，结果所有 Skill 共享
   ── ...

⑦ 组装 Response Agent 输入
```

代码骨架：

```python
class Orchestrator:
    def __init__(self, slot_manager, skill_registry):
        self.slot_manager = slot_manager
        self.skill_registry = skill_registry
    
    async def orchestrate(self, planner_output: PlannerOutput) -> ResponseInput:
        # 1. 更新 Slot
        self.slot_manager.apply(planner_output.slot_ops)
        slots = self.slot_manager.session_slots
        
        # 2. 遍历 sub_tasks
        completed = []
        pending = []
        all_tools = set()
        all_instructions = []
        
        for task in planner_output.sub_tasks:
            skill = self.skill_registry.get(task)
            if skill is None:
                continue  # 未知 sub_task 跳过
            
            # 检查 slot
            missing = [s for s in skill.required_slots if s not in slots or slots[s] is None]
            if missing:
                pending.append({"skill": task, "missing": missing})
                continue
            
            # 检查 emotion
            if not skill.check_emotion(planner_output.emotion):
                pending.append({"skill": task, "reason": "emotion"})
                continue
            
            completed.append(task)
            all_tools.update(skill.required_tools)
            all_instructions.append(skill.instruction)
        
        # 3. 统一执行 Tool
        knowledge = []
        if Tool.RAG in all_tools:
            query = self._build_query(planner_output, slots)
            knowledge = await rag.search(query)
        
        # 4. 组装 Response Agent 输入
        return ResponseInput(
            instructions=all_instructions,
            knowledge=knowledge,
            slots=slots,
            emotion=planner_output.emotion,
            completed=completed,
            pending=pending,
        )
```

### 3. Tool Layer（纯代码）

**职责**：为 Response Agent 提供执行所需的资源。由 Orchestrator 统一调度，多个 Skill 共享同一次执行结果。

#### RAG（当前唯一的 Tool）

RAG 执行链路（**已有代码不动，调用方式调整为 Orchestrator 统一触发**）：

```
Query Rewrite 输入
  ├── user_query（原始用户输入）
  ├── sub_tasks（来自 Planner）
  └── slots（来自 Session Slot Store）
          │
          ▼
  Query Rewrite
  利用 sub_tasks + slots 构造高质量检索 query
  例：用户说"多少钱"，Planner 识别 model=M8、sub_task=PRICE
  → 改写为 "M8 官方指导价"

  MemoryManager 的上下文（历史 + 摘要 + 用户画像）
  已在 Planner 阶段消费完毕，无需重复传入
          │
          ▼
  Hybrid Search（已有）
  ├── Vector（ChromaDB + all-MiniLM-L6-v2）
  └── BM25（jieba + 倒排索引改进版）
          │
          ▼
  RRF Fusion（K=60）
          │
          ▼
  TopK Chunks → 传给 Response Agent
```

**关键设计：不重复检索。** 如果同时有 PRICE + PRODUCT + FINANCE 三个 Skill，RAG 只执行一次，结果共享。

---

### 4. Response Agent（LLM）

**职责**：接收 Instruction + Context，一次生成完整回复。

输入结构：

```text
【任务 Instruction】
1. 回答用户价格问题（根据 Knowledge 回答，没有答案不要编造）
2. 回答用户产品问题（根据 Knowledge 回答）
3. 安抚用户情绪
4. 当前不适合进行留资

【Knowledge】
chunk1: M8 官方指导价 19.98-22.98 万元...
chunk2: M8 搭载 2.0T 发动机...

【Slots】
model = M8
budget = 20万

【Emotion】
negative

【History / User Profile】
（由 MemoryManager 拼接）
```

Response Agent 只知道自己要完成什么任务，不关心背后有什么 Skill。所有 Skill 的 `instruction` 已经被 Orchestrator 合并到"任务"部分。

**优势**：
- 回复风格天然一致（只有一个 LLM 生成回复）
- 不存在多 Agent 拼接时的语气冲突
- 知识引用来源统一，不会前后矛盾

## Skill 定义

**Skill 是纯元数据（Metadata-only），没有 `execute()` 方法。** 真正的执行者是 Response Agent。

```python
class BaseSkill:
    """所有 Skill 的基类 — 纯元数据"""
    
    name: str                       # 标识，与 sub_task 对应
    required_slots: list[str]       # 必需的槽位列表
    required_tools: list[Tool]      # 需要的工具列表
    instruction: str                # 给 Response Agent 的指令
    
    @classmethod
    def check_emotion(cls, emotion: str) -> bool:
        """情绪条件检查（默认通过，子类可覆盖）"""
        return True
```

### 初始 Skill 清单

| Skill | sub_task | 必需槽位 | 需要 RAG | 情感限制 | 指令摘要 |
|-------|----------|---------|---------|---------|---------|
| ProductSkill | PRODUCT | `model` | ✅ | 无 | 根据 Knowledge 回答产品问题 |
| PriceSkill | PRICE | `model` | ✅ | 无 | 根据 Knowledge 回答价格问题 |
| FinanceSkill | FINANCE | `model`, `budget` | ✅ | 无 | 提供金融方案建议 |
| ComplaintSkill | COMPLAINT | `issue` | 可选 | 无 | 先安抚情绪，不要推销 |
| LeadCaptureSkill | LEAD_CAPTURE | `phone` | ❌ | 情绪差时不可执行 | 情绪正常时引导留资，否则跳过 |
| GreetingSkill | GREETING | （无） | ❌ | 无 | 问候用户，简单互动 |

### 未来扩展 Skill 示例

```python
class TradeInSkill(BaseSkill):

    required_slots = ["old_model", "old_year"]

```

1. 新建 `skills/trade_in.py`，定义元数据
2. 注册到 `SkillRegistry`
3. Planner Prompt 里加一条 few-shot 样例
4. **Orchestrator 和 Response Agent 完全不用改**

## 面试问题预设

### Q: 你这个架构解决了什么问题？

> 之前是"一个意图路由到一个 Agent"的简单映射，但真实用户往往一句话里有多个诉求（投诉+询价+留资）。多 Agent 路由要么只能选一个处理，要么顺序执行多个 Agent，带来回复风格不一致和 Token 浪费。新架构把系统拆成"理解、编排、生成"三层：LLM 只做理解和生成，中间的编排和工具调用由程序确定性执行，一个 Response Agent 一次完成所有回复，天然保证风格一致。

### Q: Slot 为什么用 Diff 而不是全量？

> 多轮后槽位可能累积十几个字段，如果让 LLM 每轮全量输出，它漏掉一个你无法判断是故意删除还是生成遗漏。Diff 模式让 LLM 只说"这轮变化了什么"，状态由程序确定性维护，职责分离更干净。

### Q: 知识检索（RAG）在架构中是什么角色？

> RAG 在整个架构中是一个 Tool，不是 Skill。由 Orchestrator 统一调度——如果编排发现 PriceSkill 和 ProductSkill 都需要查知识库，RAG 只执行一次，结果共享。检索时会结合 Planner 输出的 sub_tasks 和 slots 做 Query Rewrite，比如用户只说"多少钱"，但 Planner 已经提取了 model=M8，改写为"M8 官方指导价"再检索，召回质量更高。

### Q: 情绪在架构中扮演什么角色？

> 情绪不作为路由信号，而是由 Orchestrator 代码层直接判断其影响——比如用户情绪差时不追问留资。这个决策放在程序层而非 LLM 层，保证行为确定性。

### Q: Skill 为什么没有 execute() 方法？

> 因为我们设计下来发现，真正"执行"业务逻辑的始终只有一个组件——Response Agent（LLM）。Skill 的本质不是执行单元，而是能力元数据：它告诉 Orchestrator 自己需要什么槽位、需要什么工具（如 RAG）、以及希望 Response Agent 完成什么任务（instruction）。这种设计让新增能力变得非常简单——声明式定义，不涉及执行逻辑。

### Q: 新增一个能力需要改多少代码？

> 只需新建一个 Skill 文件声明元数据（required_slots、required_tools、instruction），然后在 Planner Prompt 里加一条 few-shot 样例。Orchestrator 和 Response Agent 完全不用改。

### Q: 为什么不用 LangGraph 等框架？

> 我们的场景是客服对话，不是复杂多步工作流。三层架构（理解 → 编排 → 生成）加上 Skill 元数据模式已经足够覆盖所有场景，且比 LangGraph 更轻量。每层职责清晰：Planner 理解用户、Orchestrator 编排 Skill、Response Agent 生成回复，出现问题可以快速定位到具体层。引入框架反而会增加不必要的抽象和调试成本。
