# EchoMind 智能客服系统 — 简历用项目描述

> 以下内容均基于项目实际代码，不夸大，可在面试中自圆其说。

---

## 一句话定位

基于 Claude API 的智能客服系统，核心是 **多 Agent 路由编排 + 三路融合意图识别 + RAG 检索优化**。

---

## 项目亮点的 5 条表达（每条含实现 + 方案 + 可追问点）

### 1. RAG 检索优化链路：查询改写 + 混合检索 + RRF 粗排 + LLM 重排

**项目实际做了：**
- 查询改写：LLM 把用户问题扩写成多角度子查询（如"退款流程"→"如何申请退款"+"退款需要几天"+"退款政策是什么"）
- 并行召回：多个子查询同时检索 ChromaDB，合并去重
- LLM 重排：对候选集按语义相关性重新打分排序

**方案中但未实现的（面试可补充）：**
- BM25 关键词检索：弥补向量检索对专有名词、错误码匹配不精准的问题
- RRF 粗排：在合并去重后先用 RRF 做一次轻量排序，再送 LLM 精排
- 父子文档切片：检索时子块命中多个同属一个父块则返回父块，保证上下文完整

**可追问的问题：**
- 为什么向量检索不够？→ 退款政策和积分政策都有"期限"语义，向量距离近但用户要的不是同一个东西
- RRF 解决了什么问题？→ 多路召回分数量纲不同不能直接比较，RRF 按排名融合不看分数
- 重排为什么用 LLM 不用专门模型？→ 项目不想引入额外依赖，同一套 API 也能做

### 2. 三路融合意图识别

**项目实际做了：**
- LLM 语义理解（权重 70%）：Few-shot + 对话历史上下文，调 Claude API 做分类
- Embedding 向量匹配（权重 20%）：n-gram 哈希向量做余弦相似度，无外部依赖
- 关键词模式匹配（权重 10%）：零延迟硬匹配兜底
- 三路用 asyncio.gather 并行执行，耗时取 max 不累加
- 加权投票合并 + 阈值过滤（0.5），模糊不清的统一归 OTHER
- Embedding 服务不可用时自动降级为二路（LLM 85% + 关键词 15%）

**可追问的问题：**
- LLM 挂了怎么办？→ Embedding 或关键词兜底，不会整条链路断掉
- Embedding 为什么用哈希不用模型？→ 不想引入本地模型/外部 API，这路只有 20% 权重
- 阈值 0.5 怎么定的？→ 经验值，可根据误分类成本动态调整

### 3. 三级记忆管理 + 自动压缩

**项目实际做了：**
- 工作记忆（Redis）：当前会话最近 20 条消息，TTL 24h
- 情景记忆（ChromaDB）：压缩后的历史对话，支持跨会话语义检索
- 用户画像（ChromaDB）：LLM 提炼的用户偏好，异步更新不阻塞主流程
- 超过 15 条时自动压缩：LLM 生成摘要存 Redis + ChromaDB，Redis 只保留最近 5 条

**可追问的问题：**
- 为什么压缩阈值是 15？→ 超过之后 context 太长 LLM 延迟明显上升，保留 5 条保证最近对话可用
- 摘要丢了怎么办？→ Redis 有 TTL，摘要是拼接累存的，丢了也能从 ChromaDB 情景记忆检索回来

### 4. Monitor 闭环反馈影响路由

**项目实际做了：**
- 每 10s 采集所有 Agent 的成功率、平均延迟
- 成功率 < 90% 或延迟 > 3s 的 Agent 计算 0-0.9 的惩罚系数
- 惩罚值写回 Orchestrator，_best_agent() 选 Agent 时自动绕开降权实例
- 异常检测（Z-score 滑动窗口）和阈值告警

**可追问的问题：**
- 惩罚怎么算的？→ 两部分：成功率每低 1% 加一定惩罚 + 延迟每超标一定量加一定惩罚，上限 0.9
- 熔断和降级的区别？→ 熔断是工具级别的（连续失败 5 次后拒绝请求），降级是整个 Agent 路由权重调整

### 5. LLM-as-Judge 端到端评测

**项目实际做了：**
- 意图识别评测：8 条标注用例，算准确率和 Macro-F1
- 对话质量评测：5 条对话用例（含多轮），真实调 Orchestrator 生成回答后 LLM 从相关性/准确性/完整性/有用性四维打分
- 回归检测：与历史基线对比，退化超 5% 标记
- 基于规则生成优化建议

**方案中但未实现的（面试可补充）：**
- RAG 分层评估：当前只有端到端综合分，没有分开测检索质量和生成忠实度
- 接入 Langfuse 做全链路 trace：用户 query → 改写子查询 → 召回 chunk → 重排结果 → LLM 生成，每一步可回溯

**可追问的问题：**
- LLM 给自己打分有偏差怎么办？→ 训练数据用人工标注校准，定期抽检
- 多轮评测怎么做的？→ 模拟真实对话，第一轮的回答作为第二轮的历史上下文传进去，逐轮评分

## 架构图（自绘版）

```
用户 → FastAPI → Memory(Redis+ChromaDB) → RAG检索 → IntentRecognizer(三路)
    → Orchestrator(路由+降级) → Agent(调LLM) → 持久化 → 返回
                              ↑
                         Monitor(每10s反馈)
```

## 面试官可能的追问及应答

| 问题 | 应答 |
|------|------|
| 分三个 Agent 的意义？ | 不同 system_prompt 在少样本下回答风格有差异，也为后续挂不同工具预留扩展性 |
| 为什么叫 Agent 不直接叫分类器？ | 每个 Agent 有自己的 system_prompt + 状态统计 + 路由权重，是独立执行单元 |
| 路由怎么做的？ | 意图映射到 Agent 类型 → 同类型多实例按 routing_score 择优 → 专属 Agent 失败降级到 General |
| RAG 查不到怎么办？ | 有降级 fallback 返回提示文本，熔断器防雪崩，连续失败 5 次后断开 |
| 监控有什么实际效果？ | 某个 Agent 成功率下降后自动被路由绕开，负载降低后有机会恢复，不需要人工介入 |
| 评测跑一次多久？ | 8 条意图 + 5 条对话约调用 20 次 LLM，几秒到十几秒 |

## 一句话总结

> 这个项目的核心是**意图识别三路互备**、**RAG 检索多阶段优化**、**监控路由闭环自愈**。架构完整但实现克制，在简单和工程可用之间取了平衡。面试可以围绕这三条主线展开。
