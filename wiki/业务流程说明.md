# EchoMind 业务流程说明

本文档从业务视角说明 EchoMind 如何处理一次客服请求：什么时候切换 Agent，什么时候并行协作，什么时候压缩会话，什么时候更新用户画像，什么时候触发升级或降级。

## 1. 一次用户请求的完整流程

用户调用 `/chat` 或在 CLI 输入消息后，系统按下面顺序处理：

```text
用户消息
  -> 读取记忆上下文
     - Redis 当前会话最近消息
     - ChromaDB 相关历史摘要
     - ChromaDB 用户画像
  -> 意图识别
     - LLM 语义理解
     - Embedding 相似度
     - Pattern 关键词兜底
  -> Agent 路由
     - 普通问题 -> GeneralAgent
     - 技术问题 -> TechnicalAgent
     - 账单/账户问题 -> BillingAgent
     - 转人工/紧急问题 -> 升级标记
  -> Agent 生成回复
  -> 写入工作记忆
  -> 异步更新用户画像
  -> 返回给用户
```

## 2. 什么时候会切换 Agent

EchoMind 并不是固定使用一个 Agent，而是根据用户问题动态选择。

### 2.1 技术问题切到 TechnicalAgent

当意图识别结果是 `technical`，或者问题中包含明显技术关键词时，会切换到技术 Agent。

典型输入：

```text
应用一直报 401 错误
网页打不开
系统崩溃了
出现 500 error
无法登录
```

常见关键词：

```text
崩溃、报错、error、crash、无法登录、登录失败、500、401
```

业务效果：

```text
GeneralAgent -> TechnicalAgent
```

TechnicalAgent 会更倾向于给出排障步骤、错误原因和下一步处理建议。

### 2.2 账单/退款/账户问题切到 BillingAgent

当意图识别结果是 `billing` 或 `account`，或者问题包含账单关键词时，会切换到账单 Agent。

典型输入：

```text
我要退款
为什么重复扣款了？
帮我开发票
我要取消订阅
修改我的邮箱
```

常见关键词：

```text
退款、扣款、发票、账单、支付、订阅、refund、invoice
```

业务效果：

```text
GeneralAgent -> BillingAgent
```

BillingAgent 会更关注退款、发票、订阅、账户资料等问题。

### 2.3 普通咨询使用 GeneralAgent

如果问题没有明确落到技术或账单领域，会默认使用 GeneralAgent。

典型输入：

```text
你好
你们支持哪些功能？
订单什么时候到？
怎么联系客服？
```

业务效果：

```text
GeneralAgent
```

GeneralAgent 负责通用问答和简单引导。

### 2.4 转人工或高紧急度会触发升级

如果用户明确要求转人工，或者问题被识别为紧急，会触发升级标记。

典型输入：

```text
我要转人工
我要投诉
找你们经理
非常紧急，马上处理
```

业务效果：

```text
escalated = true
```

当前项目里升级是标记状态；生产系统中可以在这个位置接入工单系统、人工客服队列或告警通知。

## 3. 什么时候会多 Agent 并行协作

如果一句话同时涉及多个业务领域，系统会自动并行调用多个 Agent。

### 3.1 技术 + 账单复合问题

典型输入：

```text
登录报错 401，而且这个月还重复扣款了
```

系统会识别到：

```text
技术问题：登录报错 401
账单问题：重复扣款
```

业务处理：

```text
TechnicalAgent + BillingAgent 并行处理
```

返回形式类似：

```text
[technical]
针对 401 登录错误的排查步骤...

[billing]
针对重复扣款的处理建议...
```

### 3.2 为什么要并行而不是只选一个 Agent

如果只选 TechnicalAgent，账单问题可能被忽略；如果只选 BillingAgent，登录问题可能无法解决。

并行协作的价值：

| 业务问题 | 并行协作的收益 |
|----------|----------------|
| 一个问题跨多个领域 | 避免漏答 |
| 用户一次性提出多个诉求 | 分领域处理，更完整 |
| 专属 Agent 能力不同 | 每个 Agent 回答自己擅长的部分 |

## 4. 什么时候会降级到 GeneralAgent

降级是系统可靠性策略。即使专属 Agent 出问题，也尽量给用户一个通用答复。

会降级的情况：

| 情况 | 结果 |
|------|------|
| 目标 Agent 类型没有可用实例 | 使用 GeneralAgent |
| TechnicalAgent 执行失败 | 降级到 GeneralAgent |
| BillingAgent 执行失败 | 降级到 GeneralAgent |
| LLM 调用异常 | 返回通用错误提示 |

业务效果：

```text
专属 Agent 失败 -> GeneralAgent 兜底
```

这样用户不会直接看到系统崩溃，而是收到“稍后重试”或通用处理建议。

## 5. 什么时候会话会被压缩

EchoMind 会把当前会话最近消息存在 Redis 中，叫工作记忆。

默认规则：

```text
WORKING_MAX = 20
COMPRESS_AT = 15
```

当同一个 `user_id + conv_id` 的工作记忆达到 15 条消息时，会触发压缩。

### 5.1 压缩前

Redis 中可能有很多轮消息：

```text
user: 你好
assistant: 你好，请问有什么可以帮你？
user: 我要退款
assistant: 请提供订单号
...
```

### 5.2 压缩时

系统会：

```text
旧消息 -> LLM 总结 -> 会话摘要
```

然后执行三件事：

```text
1. 摘要写入 Redis summary:{user_id}:{conv_id}
2. 摘要写入 ChromaDB episodic
3. Redis 工作记忆只保留最近 5 条
```

### 5.3 压缩后

后续对话上下文由三部分组成：

```text
会话摘要
相关历史
最近 5 条消息
```

业务价值：

| 问题 | 压缩带来的收益 |
|------|----------------|
| 对话太长 | 控制 prompt 长度 |
| 历史信息不能丢 | 用摘要保留关键信息 |
| 后续还要查历史 | 写入 ChromaDB 供语义检索 |

## 6. 什么时候会更新用户画像

每次 `/chat` 回复完成后，系统都会异步更新用户画像。

触发时机：

```text
Agent 回复完成 -> 写入本轮对话 -> 后台更新用户画像
```

画像内容示例：

```json
{
  "preferences": ["喜欢简洁回答", "经常咨询退款"],
  "entities": {
    "产品": ["会员服务"],
    "问题类型": ["账单", "登录"]
  }
}
```

业务作用：

| 用户画像信息 | 后续作用 |
|--------------|----------|
| 偏好 | 让回复更贴近用户习惯 |
| 常见产品 | 帮助 Agent 理解用户上下文 |
| 常见问题类型 | 帮助后续回复更聚焦 |

注意：用户画像更新是异步执行的，不会阻塞当前回复。

## 7. 什么时候会查询知识库

知识库主要通过 `/search` 接口演示，也可以作为 MCP 工具被 Agent 使用。

典型业务问题：

```text
退款多久到账？
订单超过 7 天还没到怎么办？
会员积分怎么算？
登录 401 怎么处理？
```

检索流程：

```text
用户查询
  -> LLM 查询改写
  -> 多个子查询并行检索 ChromaDB knowledge_base
  -> 合并去重
  -> LLM 重排
  -> 返回 Top-K
```

当知识库不可用时，会走 fallback，返回可解释的降级结果，而不是直接报错。

## 8. 什么时候 Monitor 会影响路由

Monitor 每 10 秒采集一次 Agent 和工具表现。

采集指标：

```text
Agent 成功率
Agent 平均延迟
工具成功率
工具平均延迟
工具连续失败次数
```

如果某个 Agent 成功率低或延迟高，Monitor 会计算 `monitor_penalty` 并写回 Orchestrator。

路由分数：

```text
base_score = success_rate * 0.7 + latency_score * 0.3
routing_score = base_score * (1 - monitor_penalty)
```

业务效果：

```text
表现差的 Agent -> routing_score 降低 -> 后续更少被选中
```

如果同一类 Agent 未来扩展成多个实例，这个机制可以自动选择表现更好的实例。

## 9. 常见业务场景示例

### 场景一：普通问候

```text
用户：你好
Agent：GeneralAgent
结果：普通欢迎语
```

### 场景二：技术故障

```text
用户：应用登录一直报 401
Agent：TechnicalAgent
结果：给出登录失败排查步骤
```

### 场景三：退款问题

```text
用户：我要退款，为什么还没到账？
Agent：BillingAgent
结果：说明退款审核和到账周期
```

### 场景四：技术 + 账单复合问题

```text
用户：登录报错 401，而且还重复扣款了
Agent：TechnicalAgent + BillingAgent
结果：两个 Agent 并行回复后合并
```

### 场景五：长对话

```text
用户连续对话超过 15 条消息
系统：触发工作记忆压缩
结果：旧消息变成摘要，最近 5 条继续保留
```

### 场景六：强烈投诉

```text
用户：我要投诉，马上转人工
系统：标记 escalated = true
结果：生产环境可转工单或人工客服
```

## 10. 业务规则速查表

| 触发条件 | 系统行为 |
|----------|----------|
| 普通咨询 | 使用 GeneralAgent |
| 技术关键词或 technical 意图 | 使用 TechnicalAgent |
| 退款/扣款/发票/订阅/账户问题 | 使用 BillingAgent |
| 技术 + 账单同时出现 | 多 Agent 并行协作 |
| 专属 Agent 失败 | 降级到 GeneralAgent |
| 用户要求转人工 | 标记 `escalated = true` |
| 工作记忆达到 15 条 | 压缩旧消息，保留最近 5 条 |
| 每次回复完成 | 异步更新用户画像 |
| Agent 成功率低或延迟高 | Monitor 写回降权，降低 routing_score |
| 知识库不可用 | MCP 工具返回 fallback 降级结果 |
