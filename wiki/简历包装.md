本文档用于把 EchoMind 项目包装成简历、面试和项目展示中可讲清楚的经历。建议按真实实现描述，量化数据如果没有压测或标注集支撑，应写成“设计目标”或“在内置评测集上”。

## 项目一句话描述

> EchoMind 是一个基于 LLM 的企业级智能客服系统，集成三路融合意图识别、多 Agent 路由编排、Redis + ChromaDB 三级记忆、MCP 工具调用优化、Monitor 在线降权闭环和 LLM-as-Judge 端到端评测。

## 面试高频问题及回答思路

### Q1：端到端意图识别是怎么做的？为什么用三路融合？

**回答思路**

我没有只依赖单一策略，而是做了三路融合：

- LLM 负责复杂语义理解和上下文依赖。
- Embedding 负责常见表达和模板的相似度匹配。
- Pattern 负责零延迟兜底，覆盖退款、报错、转人工等高频关键词。

官方 API 模式下权重是 LLM 70%、Embedding 20%、Pattern 10%；如果配置了第三方兼容 API，比如 DeepSeek，因为很多兼容 API 不支持 Embedding，系统会自动切换为 LLM 85% + Pattern 15%。投票得分低于阈值时会降级为 `OTHER`，减少错误路由。

**可以补充**

当前 Anthropic SDK 没有 `embeddings` 资源，所以我做了远端优先、本地字符 n-gram 哈希向量兜底。这样官方模式下 Embedding 分支仍然可运行，不会因为 SDK 缺能力导致三路融合失效。

### Q2：检索召回不好怎么优化？

**回答思路**

我把知识库检索封装成 MCP 工具，并设计了完整检索优化链路：

```text
原始查询 -> LLM 查询改写 -> 多子查询并行召回 -> 合并去重 -> LLM 重排 -> Top-K
```

查询改写解决召回不全。例如“退款流程”会被扩展成“如何申请退款”“退款需要多久”“退款政策是什么”等多个角度。并行召回控制延迟，合并去重减少重复片段。最后用 LLM 重排，因为向量相似度高不一定代表对用户最有帮助。

**可以补充**

工具框架还有 TTL 缓存、JSON Schema 参数校验、超时控制、熔断器和 fallback。知识库不可用时会返回可解释降级结果，而不是直接抛空错误。

### Q3：多轮对话记忆怎么设计？

**回答思路**

我用了三级记忆：

| 层级 | 存储 | 作用 |
|------|------|------|
| 工作记忆 | Redis | 当前会话最近消息，24h TTL |
| 情景记忆 | ChromaDB `episodic` | 压缩后的历史对话摘要，可语义检索 |
| 用户画像 | ChromaDB `user_profile` | 从对话中提炼用户偏好和关键实体 |

当前会话消息先写入 Redis。达到压缩阈值后，用 LLM 总结旧消息，把摘要写入 Redis summary 和 ChromaDB `episodic`，工作记忆只保留最近 5 条。这样既保留当前上下文，又避免 prompt 无限增长。

**可以补充**

每次 `/chat` 回复后会异步更新用户画像，不阻塞主请求。下一次同一用户请求时，画像会作为 `[用户画像]` 拼入 Agent 背景信息，用于个性化回复。

### Q4：多 Agent 路由怎么做？

**回答思路**

路由分三层：

1. 意图映射：技术问题到 `TechnicalAgent`，账单/账户问题到 `BillingAgent`，其他到 `GeneralAgent`。
2. 性能路由：同类 Agent 多实例时，按 `routing_score` 选择表现最好的实例。
3. 降级路由：专属 Agent 不可用或失败时，降级到 `GeneralAgent`。

`routing_score` 不只是成功率，还考虑平均延迟和 Monitor 写回的 `monitor_penalty`：

```text
base_score = success_rate * 0.7 + latency_score * 0.3
routing_score = base_score * (1 - monitor_penalty)
```

**可以补充**

当前主链路支持自动并行协作。如果用户说“登录报错 401，而且还重复扣款”，系统会识别到技术和账单两个领域，自动并行调用 `TechnicalAgent` 和 `BillingAgent`，再合并结果。

### Q5：Monitor 怎么形成闭环？

**回答思路**

Monitor 每 10 秒采集 Orchestrator 和 ToolManager 的统计数据，包括 Agent 成功率、平均延迟、工具成功率、工具延迟和连续失败次数。

闭环体现在它不只是展示指标，还会计算每个 Agent 的 `monitor_penalty`，然后调用：

```python
Orchestrator.update_routing_penalties()
```

把惩罚项写回 AgentStats。下一次 `_best_agent()` 选择 Agent 时，`routing_score` 会受这个惩罚项影响，表现差的 Agent 会自动降权。

### Q6：端到端 Agent 怎么评测？

**回答思路**

评测分两部分：

1. 意图识别评测：用标注用例计算 Accuracy 和 Macro-F1。
2. 端到端回复评测：真实调用 `Orchestrator.run()` 生成回复，再用 LLM-as-Judge 从四个维度打分。

四个维度是：

- `relevance`：是否回答用户问题。
- `accuracy`：信息是否准确。
- `completeness`：是否完整解决需求。
- `helpfulness`：用户能否据此行动。

评测器还会和上一轮结果对比，如果某个指标退化超过 5%，会标记为 regression，并生成优化建议。

### Q7：ChromaDB 在项目里怎么用？

**回答思路**

ChromaDB 有三个 collection：

| Collection | 用途 |
|------------|------|
| `knowledge_base` | RAG 知识库，存业务文档片段 |
| `episodic` | 情景记忆，存压缩后的历史对话摘要 |
| `user_profile` | 用户画像，存偏好和关键实体 |

这三个 collection 分工不同，避免知识库文档和用户记忆混在一起。Docker Compose 中 ChromaDB 单独运行，应用通过 `chromadb.HttpClient(host="chromadb", port=8000)` 连接。

## 30 秒项目 Pitch

> EchoMind 是一个基于 LLM 的智能客服系统。我主要解决了六个工程问题：第一，意图识别用 LLM、Embedding 和 Pattern 三路融合；第二，知识库检索用查询改写、并行召回和 LLM 重排提升质量；第三，多轮对话用 Redis 工作记忆、ChromaDB 情景记忆和用户画像管理上下文；第四，多 Agent 通过意图路由、性能路由和降级路由协作处理问题；第五，Monitor 会根据线上成功率和延迟写回降权因子，动态影响后续路由；第六，用 LLM-as-Judge 做端到端评测和回归检测。整个系统可以用 Docker Compose 一键启动。

## 面试时的注意事项

- 不要把“设计目标”说成“线上真实指标”，除非你确实跑过压测或标注评测。
- 讲 Monitor 时要强调当前实现是 `monitor_penalty` 写回 Orchestrator，不只是展示 dashboard。
- 讲并行协作时要强调当前主链路会自动检测复合问题，不需要外部手动调用 `run_parallel()`。
- 讲 ChromaDB 时要明确三个 collection 的边界：`knowledge_base`、`episodic`、`user_profile`。
- 讲 Embedding 时要说明第三方兼容 API 默认禁用，官方模式下支持远端优先、本地哈希向量兜底。
