# EchoMind 完整使用指南

本文档详细说明项目的三种部署方式、HTTP 接口调用方法、评测指标接口、以及 CLI 交互模式的使用。

---

## 一、部署方式总览

| 方式 | 适用场景 | 特点 |
|------|----------|------|
| docker compose 全栈部署 | 生产环境 / 完整体验 | 一键启动所有服务，含 Redis + ChromaDB + Prometheus |
| docker run 单容器 | 开发调试 / CI/CD | 代码挂载，改了立刻生效，不用重新 build |
| docker run --cli | 快速体验 / 演示 | 终端直接对话，不启动 HTTP 服务器 |

---

## 二、方式 A：docker compose 全栈部署

### 启动

```bash
cd EchoMind

# 编辑 .env 填写 API Key
vim .env

# 一键构建镜像 + 启动所有服务（--build 确保用最新代码构建）
docker compose up -d --build

# 查看日志（看到小熊饼干图案说明就绪）
docker compose logs -f echomind
```

> `--build` 会自动构建 `echomind` 镜像。首次构建约 2-3 分钟（安装依赖），后续有缓存会快很多。

### 启动后的服务

| 服务 | 端口 | 说明 |
|------|------|------|
| EchoMind API | 8000 | 主应用 |
| Redis | 6379 | 工作记忆存储 |
| ChromaDB | 8001 | 向量数据库 |
| Prometheus | 9090 | 监控指标 |
| Nginx | 80 | 反向代理（可选） |

### 停止

```bash
docker compose down       # 停止，保留数据
docker compose down -v    # 停止并清除所有数据
```

## 三、方式 B：docker run 单容器（HTTP 服务器模式）

### 前置：构建镜像 + 启动依赖服务

```bash
# ① 构建 echomind 镜像（首次或代码依赖变更后执行）
docker compose build echomind

# ② 启动 Redis 和 ChromaDB
docker compose up -d redis chromadb
```

> 镜像里打包了所有 Python 依赖，后续 `-v $(pwd):/workspace` 只是用本地代码覆盖容器内代码，依赖不受影响。

### 启动 HTTP 服务器

```bash
docker run -it --rm \
  --network echomind_echomind-network \
  -p 8000:8000 \
  -e ANTHROPIC_BASE_URL="https://api.deepseek.com/anthropic" \
  -e ANTHROPIC_API_KEY="sk-你的key" \
  -e ANTHROPIC_MODEL="deepseek-chat" \
  -e REDIS_URL="redis://:echomind123@redis:6379/0" \
  -e CHROMA_PERSIST_DIRECTORY="/workspace/data/chroma" \
  -v $(pwd):/workspace \
  -w /workspace \
  echomind
```

**参数解释**：

| 参数 | 作用 |
|------|------|
| `--network echomind_echomind-network` | 加入 compose 网络，可直接用 `redis` 服务名通信 |
| `-p 8000:8000` | 端口映射，宿主机可访问 |
| `-v $(pwd):/workspace` | 挂载本地代码到容器，改代码不用重新 build |
| `-w /workspace` | 容器工作目录设为挂载目录 |
| 不加末尾命令 | 默认执行 `CMD`，即启动 uvicorn HTTP 服务器 |

### 启动成功标志

终端输出：

## 四、方式 C：docker run CLI 交互模式（不启动 HTTP）

> 前置条件同方式 B：需要先 `docker compose build echomind` 构建镜像，并启动 Redis。

```bash
docker run -it --rm \
  --network echomind_echomind-network \
  -e ANTHROPIC_BASE_URL="https://api.deepseek.com/anthropic" \
  -e ANTHROPIC_API_KEY="sk-你的key" \
  -e ANTHROPIC_MODEL="deepseek-chat" \
  -e REDIS_URL="redis://:echomind123@redis:6379/0" \
  -e CHROMA_PERSIST_DIRECTORY="/workspace/data/chroma" \
  -v $(pwd):/workspace \
  -w /workspace \
  echomind \
  python api/main.py --cli
```

**与方式 B 的唯一区别**：末尾加了 `python api/main.py --cli`，覆盖了默认的 `CMD`。

### 交互示例

```
    ʕ•ᴥ•ʔ  ʕ•ᴥ•ʔ  ʕ•ᴥ•ʔ
   ╔══════════════════════╗
   ║   EchoMind  v2.0     ║
   ║   智能客服 AI 系统    ║
   ╚══════════════════════╝
    ʕ•ᴥ•ʔ  ʕ•ᴥ•ʔ  ʕ•ᴥ•ʔ

EchoMind CLI — 输入 quit 退出

你: 我的订单什么时候到？

EchoMind [general]: 您好！请提供您的订单号，我帮您查询物流状态。

你: 订单号是 #12345

EchoMind [general]: 您的订单 #12345 已发货，预计 2-3 个工作日送达...

你: quit
再见 ʕ•ᴥ•ʔ
```

**特点**：
- 直接在终端输入对话，回车发送
- 输入 `quit`、`exit` 或 `退出` 结束
- 按 `Ctrl+C` 也可退出
- 支持多轮对话记忆（通过 Redis）
- 每条回复会显示处理该消息的 Agent 类型（如 `[general]`、`[technical]`、`[billing]`）

## 五、HTTP 接口调用详解

以下所有接口在方式 A 和方式 B 启动后均可使用。

### 5.1 健康检查

```bash
curl http://localhost:8000/health
```

**响应**：
```json
{
  "status": "ok",
  "agents": {
    "general_0": {"total": 10, "success_rate": 0.95, "avg_ms": 820.3, "routing_score": 0.72},
    "technical_0": {"total": 3, "success_rate": 1.0, "avg_ms": 950.1, "routing_score": 0.78},
    "billing_0": {"total": 2, "success_rate": 1.0, "avg_ms": 780.5, "routing_score": 0.81}
  }
}
```

---

### 5.2 主对话接口

```bash
curl -X POST http://localhost:8000/chat \
  -H "Content-Type: application/json" \
  -d '{
    "message": "我的订单 #12345 还没到，已经超时了",
    "user_id": "user_001",
    "conv_id": "conv_abc"
  }'
```

**请求参数**：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `message` | string | 是 | 用户消息 |
| `user_id` | string | 否 | 用户 ID，默认 "anonymous" |
| `conv_id` | string | 否 | 会话 ID，不传则自动生成 |

**响应**：
```json
{
  "conv_id": "conv_abc",
  "response": "非常抱歉给您带来不便。我帮您查询了订单 #12345 的状态...",
  "intent": "query",
  "agent_type": "general",
  "escalated": false,
  "latency_ms": 842.3
}
```

**响应字段说明**：

| 字段 | 说明 |
|------|------|
| `conv_id` | 会话 ID，后续对话传入同一 ID 可保持上下文 |
| `response` | Agent 的回复内容 |
| `intent` | 识别出的意图（query/complaint/technical/billing 等） |
| `agent_type` | 处理该请求的 Agent 类型 |
| `escalated` | 是否触发了升级（转人工） |
| `latency_ms` | 端到端处理延迟（毫秒） |

**多轮对话示例**：

```bash
# 第一轮
curl -X POST http://localhost:8000/chat \
  -H "Content-Type: application/json" \
  -d '{"message": "我要退款", "user_id": "user_001", "conv_id": "session_1"}'

# 第二轮（传入相同 conv_id，系统会记住上下文）
curl -X POST http://localhost:8000/chat \
  -H "Content-Type: application/json" \
  -d '{"message": "订单号是 #67890", "user_id": "user_001", "conv_id": "session_1"}'
```

---

### 5.3 实时监控接口

```bash
curl http://localhost:8000/monitor
```

**响应**：
```json
{
  "agents": {
    "general_0": {"total": 15, "success_rate": 0.93, "avg_ms": 850, "routing_score": 0.71},
    "technical_0": {"total": 5, "success_rate": 1.0, "avg_ms": 920, "routing_score": 0.77}
  },
  "tools": {
    "knowledge_search": {"total": 8, "success_rate": 0.875, "avg_latency_ms": 320, "circuit_state": "closed"}
  },
  "alerts": [],
  "recommendations": ["P95 响应时间偏高，建议增加缓存 TTL"]
}
```

**字段说明**：

| 字段 | 说明 |
|------|------|
| `agents` | 每个 Agent 的实时统计（成功率、延迟、路由评分） |
| `tools` | 每个工具的调用统计（成功率、延迟、熔断器状态） |
| `alerts` | 当前活跃告警列表 |
| `recommendations` | 基于指标自动生成的优化建议 |

---

### 5.4 检索优化演示接口

```bash
curl -X POST "http://localhost:8000/search?query=退款流程&top_k=3"
```

**响应**：
```json
{
  "query": "退款流程",
  "results": [
    {"title": "关于「如何申请退款」的文档", "content": "..."},
    {"title": "关于「退款需要多少天」的文档", "content": "..."},
    {"title": "关于「退款政策是什么」的文档", "content": "..."}
  ],
  "reranked": true
}
```

**这个接口展示了完整的检索优化链路**：
1. 原始查询 "退款流程" 被改写为多个子查询
2. 子查询并行检索
3. 结果合并去重
4. LLM 重排后返回 Top-K

---

### 5.5 评测指标接口

```bash
curl -X POST http://localhost:8000/eval/run
```

**响应**：
```json
{
  "pass_rate": 0.85,
  "avg_scores": {
    "relevance": 0.88,
    "accuracy": 0.82,
    "completeness": 0.79,
    "helpfulness": 0.85
  },
  "regressions": [],
  "recommendations": [
    "意图识别准确率低于 90%，建议增加训练样本",
    "BILLING 类意图的 completeness 评分偏低，建议优化 BillingAgent 的 system prompt"
  ]
}
```

**字段说明**：

| 字段 | 说明 |
|------|------|
| `pass_rate` | 整体通过率（意图识别正确 + 响应质量达标的比例） |
| `avg_scores` | LLM-as-Judge 四维度平均分（0-1） |
| `regressions` | 与历史基线对比的退化项（退化超 5% 会列出） |
| `recommendations` | 基于评测结果自动生成的优化建议 |

**评测维度详解**：

| 维度 | 评判标准 |
|------|----------|
| relevance（相关性） | 响应是否直接回答了用户的问题 |
| accuracy（准确性） | 提供的信息是否事实正确 |
| completeness（完整性） | 是否完整解决了用户需求 |
| helpfulness（有用性） | 对用户的实际帮助程度 |

## 七、完整使用流程示例

### 场景：从零开始体验全部功能

```bash
# 1. 启动所有服务
cd EchoMind
docker compose up -d --build

# 2. 等待就绪（约 30-60s）
docker compose logs -f echomind
# 看到 "EchoMind 已就绪" 后 Ctrl+C

# 3. 健康检查
curl http://localhost:8000/health

# 4. 发起对话
curl -X POST http://localhost:8000/chat \
  -H "Content-Type: application/json" \
  -d '{"message": "你好，我想退款", "user_id": "demo", "conv_id": "s1"}'

# 5. 继续对话（多轮）
curl -X POST http://localhost:8000/chat \
  -H "Content-Type: application/json" \
  -d '{"message": "订单号 #99999，买了 3 天就坏了", "user_id": "demo", "conv_id": "s1"}'

# 6. 查看监控（看 Agent 统计）
curl http://localhost:8000/monitor

# 7. 测试检索优化
curl -X POST "http://localhost:8000/search?query=如何退货"

# 8. 运行评测
curl -X POST http://localhost:8000/eval/run

# 9. 切换到 CLI 模式体验
docker compose exec echomind python api/main.py --cli
```

## 八、三种方式对比

| 对比项 | compose 全栈 | docker run HTTP | docker run CLI |
|--------|-------------|-----------------|----------------|
| 启动命令 | `docker compose up -d` | `docker run -p 8000:8000 ...` | `docker run ... python api/main.py --cli` |
| 对话方式 | HTTP 接口 | HTTP 接口 | 终端直接输入 |
| 需要 curl | 是 | 是 | 否 |
| 多轮记忆 | 支持（Redis） | 支持（Redis） | 支持（Redis） |
| 监控面板 | 支持（/monitor） | 支持（/monitor） | 不支持 |
| 评测接口 | 支持（/eval/run） | 支持（/eval/run） | 不支持 |
| 代码热更新 | 需要 rebuild | 挂载目录，立刻生效 | 挂载目录，立刻生效 |
| 适合场景 | 生产部署 | 开发调试 | 快速演示 |
