# EchoMind 技术亮点详解

本文档深入解析 EchoMind 的六大核心技术亮点，每个亮点对应一个独立模块。

---

## 亮点一：端到端意图识别

**文件**：`core/intent_recognizer.py`

**核心问题**：如何准确理解用户的真实意图？

### 设计思路

单一策略各有短板：LLM 准确但慢，关键词快但粗糙，向量匹配居中。EchoMind 采用三路融合 + 加权投票：

```
用户消息
    ├── LLM 语义理解（权重 85%）  ← Few-shot + 对话上下文
    ├── Embedding 相似度（权重 —） ← 官方 API 可用时 20%，第三方时禁用
    └── 关键词模式匹配（权重 15%） ← 零延迟兜底

        ↓ 加权投票
    最终意图 + 置信度 + 紧急度 + 实体
```

### 关键设计点

1. **并行调用**：LLM 和 Embedding 通过 `asyncio.gather` 并行，不串行等待
2. **动态权重**：检测到第三方 API（无 Embedding 支持）时自动调整为 LLM 85% + Pattern 15%
3. **在线学习**：`learn()` 方法接收人工纠正，将新样本加入模板，清除 Embedding 缓存
4. **LRU 缓存**：相同消息直接返回缓存结果，超 1000 条时淘汰最旧一半
5. **实体提取**：用 LLM 从消息中提取订单号、产品名、错误码等结构化信息

### 支持的意图类型

| 意图 | 说明 | 示例 |
|------|------|------|
| QUERY | 查询信息 | "我的订单状态是什么？" |
| COMPLAINT | 投诉不满 | "等了好几个小时！" |
| REQUEST | 请求操作 | "帮我取消订单" |
| TECHNICAL | 技术问题 | "应用一直崩溃" |
| BILLING | 账单/退款 | "为什么扣了两次款？" |
| ACCOUNT | 账户管理 | "修改邮箱" |
| ESCALATION | 要求升级 | "转人工客服" |
| GREETING | 问候 | "你好" |
| FEEDBACK | 正面反馈 | "服务很棒！" |

## 亮点二：MCP 工具调用框架

**文件**：`mcp/tool_manager.py`

**核心问题**：工具调用出错（检索不全、召回不好）怎么优化？

### 设计思路

```
用户查询: "退款流程"
    ↓
1. 查询改写（Query Rewriting）
   LLM 生成多角度子查询:
   ["如何申请退款", "退款需要多少天", "退款政策是什么"]
    ↓
2. 并行召回
   3 个子查询同时检索 → 合并去重
    ↓
3. LLM 重排（Reranking）
   按语义相关性重新打分排序 → 取 Top-K
    ↓
4. 返回高质量结果
```

### 解决的两个核心问题

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 召回不全 | 单一查询只能覆盖一个角度 | 查询改写：LLM 生成 3 个不同角度的子查询，并行检索后合并 |
| 召回不好 | 向量相似度 ≠ 对用户有用 | LLM 重排：用 LLM 理解语义相关性，重新排序 |

### 可靠性保障

- **熔断器**：连续失败 5 次自动断开，60s 后探测恢复
- **TTL 缓存**：相同参数直接返回缓存，减少重复调用
- **降级策略**：工具不可用时返回有意义的降级结果
- **超时控制**：每个工具独立超时配置

## 亮点三：多轮对话记忆管理

**文件**：`memory/conversation_memory.py`

**核心问题**：如何管理多轮对话的记忆，防止 context 爆炸？

### 三级记忆架构

```
┌─────────────────────────────────────────────┐
│  工作记忆（Redis）                            │
│  当前会话最近 20 条消息，毫秒级读写            │
│  TTL: 24h                                    │
└─────────────────────────────────────────────┘
        ↓ 超过 15 条时自动压缩
┌─────────────────────────────────────────────┐
│  情景记忆（ChromaDB）                         │
│  历史对话的 LLM 摘要，按语义向量检索           │
│  跨会话关联                                   │
└─────────────────────────────────────────────┘
        ↓ 从对话中提炼
┌─────────────────────────────────────────────┐
│  用户画像（ChromaDB）                         │
│  偏好、常用实体、行为模式                      │
│  持久化存储                                   │
└─────────────────────────────────────────────┘
```

### 自动压缩机制

```
工作记忆达到 15 条
    ↓
LLM 对旧消息生成 2-3 句摘要
    ↓
摘要存入 Redis（覆盖旧摘要）
旧消息存入情景记忆（ChromaDB）
    ↓
工作记忆只保留最近 5 条
```

### 上下文构建

每次请求时，三级记忆融合为一个 `MemoryContext`：

```python
[会话摘要]
之前讨论了退款问题，用户对处理速度不满意。

[相关历史]
- 用户曾在 3 天前咨询过同一订单
- 上次对话中提到希望加急处理

[用户画像]
{"preferences": ["快速响应"], "entities": {"产品": ["手机壳"]}}

[最近对话]
user: 我的退款到了吗？
assistant: 让我帮您查询...
```

## 亮点四：多 Agent 路由与编排

**文件**：`agents/agent_orchestrator.py`

**核心问题**：多 Agent 情况下如何做 Routing？

### 三层路由决策

```
用户请求
    ↓
┌─ 第一层：意图映射路由 ─────────────────────┐
│  TECHNICAL  → TechnicalAgent               │
│  BILLING    → BillingAgent                 │
│  ESCALATION → 直接升级                      │
│  其他       → GeneralAgent                 │
└────────────────────────────────────────────┘
    ↓
┌─ 第二层：性能路由（同类多实例时）──────────────┐
│  routing_score = 成功率 × 0.7 + 低延迟分 × 0.3│
│  选得分最高的实例                             │
└────────────────────────────────────────────┘
    ↓
┌─ 第三层：降级路由 ─────────────────────────┐
│  专属 Agent 失败 → 自动降级到 GeneralAgent  │
└────────────────────────────────────────────┘
```

### 并行协作

复杂问题（如同时涉及技术和账单）可并行派发：

```python
result = await orchestrator.run_parallel(req, [AgentType.TECHNICAL, AgentType.BILLING])
# 两个 Agent 同时处理，结果合并后返回
```

### 升级机制

- Agent 响应中包含"转人工"等关键词 → 自动标记升级
- 用户紧急度为 CRITICAL → 直接升级
- 生产环境可对接工单系统

## 亮点五：Monitor 监控 Agent 在线表现

**文件**：`monitor/performance_monitor.py`

**核心问题**：如何利用 Monitor 监控 Agent 的在线表现？

### 闭环反馈机制

```
Orchestrator 处理请求
    ↓ 每次请求后实时更新 AgentStats
Monitor 定时采集（默认 10s）
    ↓ 聚合指标 + Z-score 异常检测
发现某 Agent 成功率下降
    ↓ 触发告警（日志 + 可选 Webhook）
Orchestrator._best_agent() 读取 routing_score
    ↓ 问题 Agent 的 routing_score 自动降低
路由权重动态调整 ← 闭环完成
```

### 监控指标

| 指标 | 来源 | 用途 |
|------|------|------|
| Agent 成功率 | AgentStats | 路由决策 + 告警 |
| Agent 平均延迟 | AgentStats | 路由决策 |
| 工具调用成功率 | ToolStats | 熔断判断 |
| 工具平均延迟 | ToolStats | 优化建议 |
| 异常检测 | Z-score | 自动告警 |

### 优化建议自动生成

Monitor 基于指标自动生成可操作的优化建议：

- P95 响应时间 > 3s → 建议增加缓存 TTL
- 错误率 > 3% → 建议审查 Agent 路由规则
- 某工具连续失败 → 建议检查降级策略

## 亮点六：端到端 Agent 评测

**文件**：`evaluation/evaluator.py`

**核心问题**：如何评测端到端 Agent？

### LLM-as-Judge

用 LLM 作为评判者，从四个维度给 Agent 响应打分：

| 维度 | 说明 | 评分范围 |
|------|------|----------|
| 相关性 | 响应是否直接回答了用户问题 | 0-1 |
| 准确性 | 信息是否事实正确 | 0-1 |
| 完整性 | 是否完整解决了用户需求 | 0-1 |
| 有用性 | 对用户的实际帮助程度 | 0-1 |

### 评测流程

```
1. 意图识别评测
   输入标注数据 → 预测 vs 标注 → Accuracy / F1

2. 端到端对话评测
   模拟多轮对话 → LLM-as-Judge 打分 → 平均质量分

3. 回归测试
   当前结果 vs 历史基线 → 退化超 5% 自动标记

4. 生成优化建议
   基于评分自动输出可操作建议
```

### 使用方式

```bash
# 通过 API 触发评测
curl -X POST http://localhost:8000/eval/run

# 返回
{
  "pass_rate": 0.85,
  "avg_scores": {"relevance": 0.9, "accuracy": 0.85, ...},
  "regressions": [],
  "recommendations": ["意图识别准确率低于 90%，建议增加训练样本"]
}
```

## 模块间协作关系

```
用户请求
    ↓
[api/main.py] 接收请求
    ↓
[memory] 读取三级记忆上下文
    ↓
[intent_recognizer] 三路融合识别意图
    ↓
[agent_orchestrator] 三层路由选 Agent → 执行 → 降级
    ↓ Agent 内部可能调用
[mcp/tool_manager] 查询改写 → 并行召回 → 重排
    ↓
[memory] 写入本轮对话
    ↓
[monitor] 实时更新统计 → 影响下次路由
    ↓
返回响应
```
