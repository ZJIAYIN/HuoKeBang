## 项目介绍话术

### ❌ 之前的回答（问题所在）

> "这个项目的本质是一个引导留资型的智能客服助手，他的使用场景就是当用户进行选车购车的时候会提出问题，获客帮的目的就是从用户的提问中分析意图以及情绪，分析用户是否是潜在客户。如果是的话就要在后续的回答中引导客户进行留资，并且既然是智能客服对于用户提问，rag链路这块我也有深入的思考。。。。"

**问题分析：**
- 开头冗长绕口（"这个项目的本质是..."）
- 逻辑跳跃：意图 → 留资 → RAG，没有主线
- "rag链路这块我也有深入的思考"——太被动，像在找补
- 没有体现"我做了什么"

---

### ✅ 推荐结构：一句话定性 → 核心链路 → 两个深度亮点 → 闭环收尾

#### 第 1 段：一句话定性（15 秒）

> "这个项目是一个**留资型智能客服系统**，用户在看车选车过程中提出各种问题，系统要实时判断对方是不是潜在客户，是的话就在回答中自然引导留资，同时把对话数据沉淀下来持续优化模型。"

**为什么好：** 一句话把"是什么、给谁用、解决什么问题"全说清楚了。

---

#### 第 2 段：核心链路（30 秒）

> "整体流程分四步：
> **第一步**是意图识别——用户发一句话，我们同时判断他的意图和情绪，比如他是来询价的、还是投诉的，情绪是积极的还是犹豫的；
> **第二步**根据意图路由到不同的 Agent——询价的走 ConsultAgent 做专业解答，给联系方式了就走 LeadCaptureAgent 确认存储，情绪差就走 EscalationAgent 安抚；
> **第三步**是 RAG 检索——涉及业务知识的提问，会走查询改写、混合检索、重排这条链路从知识库里找答案；
> **第四步**是留资决策——结合意图和情绪判断当前适不适合引导留资，同时更新用户画像供后续对话使用。"

**为什么好：** 用"第一步→第二步→第三步→第四步"串起来，逻辑清晰，面试官能跟上。

---

#### 第 3 段：深度亮点（60 秒）—— 选 1-2 个展开

**亮点 A：意图识别的二维矩阵**

> "我重点做了意图识别这块。传统的客服系统只分意图，但留资场景里情绪同样关键——用户说'太贵了'，光知道他是询价不够，还得知道他是在犹豫还是在嫌贵。所以我设计了一个**意图×情绪的二维矩阵**：9 种意图 × 5 种情绪，LLM 一次输出两个标签。然后拿这个矩阵去做留资时机决策——比如 PURCHASE + POSITIVE 就一定要引导，SKEPTICAL 就先答疑不急着要联系方式。转人工也是看这个矩阵触发的。"

**亮点 B：检索链路的优化**

> "RAG 这边我搭了一套完整的检索优化链路。最核心的问题是单次检索召回不全——比如用户问'退款流程'，向量检索可能只召回语义相近的文档，但 BM25 可以精确命中'退款'这个关键词。所以我的方案是**混合检索**：向量 + BM25 走 RRF 融合排序，上层再用 LLM 做二次重排。另外还做了一个**查询改写**——把用户的一个问题扩写成三四个不同角度的子查询并行召回，召回率提升很明显。"

**亮点 C：评测体系 + 数据闭环**

> "我觉得光做功能不够，得能证明它有效。所以我搭了一套端到端的评测体系：先离线跑 Golden Test Set 算准确率和 Macro-F1（363 条人工标注的用例），线上对话则用 LLM-as-Judge 从相关性、准确性、完整性、有用性四个维度打分。更关键的是**数据闭环**——线上每一条预测结果都持久化下来，抽 Bad Case 人工修正后补充到训练集里，为后续本地模型微调做准备。"

---

#### 第 4 段：收尾（5 秒）

> "整体上就是意图理解做精、检索做全、效果可量化、数据能回流。"

## 记忆模块（短期记忆 + 长期记忆）

### 面试话术

> **"记忆模块我设计了一个三级架构，类比人的记忆机制——工作记忆、情景记忆和用户画像。** 为什么要分三层？因为客服场景其实面临一个矛盾：既要记住当前聊到哪了，又要能跨会话识别老用户，同时还得控制 prompt 长度不然模型会跑偏。
>
> **短期记忆我用 Redis。** 每次对话的消息都 lpush 到 Redis 列表里，最多保留 20 条，key 按 `wm:用户ID:会话ID` 组织，TTL 24 小时。但这还不够——一个会话可能聊几十轮，全塞进 prompt 肯定不行。所以我设了一个**压缩阈值**：当积累到 15 条时触发自动压缩——把前面的消息用 LLM 总结成 2-3 句话的摘要存到 Redis 里，Redis 里只保留最近 5 条继续对话。这样 prompt 里拿到的永远是精简版。
>
> **长期记忆我用了 ChromaDB。** 压缩产生的摘要不只是存 Redis，还会独立写入 ChromaDB 的 `episodic` 集合——每次压缩一条，不做合并。这样当用户隔几天开新会话回来，系统可以用当前问题做语义搜索，从 ChromaDB 里召回之前聊过的相关内容放到 prompt 里，不用从头问起。另外还有独立的 `user_profile` 集合，专门存 LLM 从对话中提炼的用户偏好和实体，覆盖更新。
>
> **选 Redis+ChromaDB 而不是全放 Redis 或全放数据库，是因为两个场景对不上。** 短期记忆要求低延迟、自动过期，Redis 天然适合；长期记忆要求持久化、语义检索，ChromaDB 的向量搜索更适合。分层的核心收益是：压缩机制让 prompt 不爆炸，跨会话召回让体验有连续性，用户画像让 Agent 回复更有针对性。"

### 可能的追问

**Q：为什么不用纯 Redis？**
> 单条存没问题，但要做"找相关历史"得全量扫描，代价太高。ChromaDB 的向量语义搜索可以做到按内容相似度匹配查询。Redis 只存当前会话的摘要，ChromaDB 存所有历史的摘要，各管各的。

**Q：Redis 摘要和 ChromaDB episodic 里都存了摘要，不是重复吗？**
> 形式上都是摘要，但角色不同。Redis 的是**累积式**的——每次压缩追加到旧摘要后面，变成一个越来越长的完整会话记录，直接给 LLM 作上下文。ChromaDB 的是**独立式**的——每条摘要单独一条记录，不做合并，目的是按需召回。打个比方：Redis 摘要是你手边笔记本上的会议纪要，ChromaDB 是归档到文件柜里的历史记录，需要时去翻。

**Q：如果 Redis 挂了怎么办？**
> Redis 存的是工作记忆和当前会话摘要，挂了最坏情况是当前会话没有上下文，但 ChromaDB 里的历史摘要是持久化的，下次对话时能搜回来，不会完全失忆。Redis 本身在 docker-compose 里配了 AOF 持久化，重启可恢复。

**Q：Episodic 的检索效果怎么保证？**
> 每条摘要就 2-3 句话，语义比较聚焦，检索命中率还可以。ChromaDB 用的是内置的 all-MiniLM-L6-v2 模型做 embedding，我们没有额外维护向量模型。实际线上数据我们会持久化到 intent_logs.jsonl 里做后续分析，发现 Bad Case 再针对性优化。

## 意图识别

### 面试话术

> **"意图识别这块我核心的思路是：不单单让大模型判断当前这一句话，而是尽量提供完整的上下文语义和参考信息。** 传统的意图分类只传当前 query，但在对话场景里脱离上下文很容易误判。比如用户只说"那算了"——单看这句话看不出意图，但结合前面聊的产品和价格就能知道他是放弃购买还是在压价。
>
> 我的做法是把三个信息拼在一起送给大模型：
> **第一是工作记忆里的多轮会话**——从 Redis 取出最近的消息历史，让模型知道对话上下文；
> **第二是当前的 query**——用户这次具体说了什么；
> **第三是内置的 few-shot 示例**——每个意图和情绪标签都给了一个典型例子，避免模型自由发挥。
>
> 大模型一次输出两个标签：**意图 × 情绪**。意图有 9 种（询价、咨询参数、投诉、比价等），情绪有 5 种（正面、中性、犹豫、焦虑、负面）。
>
> 拿到这个二维矩阵后，下游再去做两件事：**一是路由到对应的角色 Agent**——比如询价的走 ConsultAgent，投诉的走 EscalationAgent；**二是留资时机判断**——PURCHASE + POSITIVE 就一定引导，SKEPTICAL 就先答疑不急着要联系方式。这样整个链路就是从理解到决策串起来了。"

### 可能的追问

**Q：为什么不用传统的分类模型（BERT/textCNN）而是用大模型？**
> 一开始考虑过，但意图×情绪的组合有 45 种，传统模型需要大量标注数据，而我们初期没有这个条件。大模型零样本能力就能做到，而且上下文窗口可以容纳多轮对话，准确率够用。后续 intent_logs.jsonl 积累起来后，可以作为训练数据微调一个本地小模型来降本。

**Q：你的 few-shot 示例怎么设计的？**
> 每个意图和情绪标签配了一个典型对话片段作为例子。比如 INQUIRY_PRICE 的示例是"这款车落地多少钱？"，SENTIMENT_SKEPTICAL 的示例是"网上都说这车毛病多"，放在 system prompt 里。

**Q：如果有多个意图怎么处理？**
> 目前是单标签分类，取置信度最高的。实际场景里一个 query 确实可能包含多个意图（比如问价格的同时问配置），后续迭代方向可以是多标签输出，或者把"咨询参数"和"比价"这类常见组合作为复合意图处理。

## 留资时机决策

### 面试话术

> **"留资时机的判断我基于意图×情绪的二维矩阵来做，不是简单地看用户说了什么就推联系方式。** 核心逻辑在 `_should_ask_contact()` 这个方法里，三层过滤：
>
> **第一层：意图过滤。** 只有价格咨询（PRICE_INQ）、购买意向（PURCHASE）、产品咨询（PRODUCT_INQ）这三种意图才会考虑引导留资。像投诉、比价、对比竞品这些场景直接过滤掉，不骚扰用户。
>
> **第二层：情绪过滤。** 悲观（NEGATIVE）、焦虑（ANXIOUS）、犹豫（SKEPTICAL）的情绪也不会引导——用户都不爽了还追着要电话，体验太差。只有当情绪是正面或中性时才引导。特别是 PURCHASE + POSITIVE 这个组合，代码里是硬编码的，直接标记为必须引导。
>
> **第三层：拒绝冷却窗口。** 用户如果明确说了"暂时不想留"、"再说吧"——意图识别会识别为 CONTACT_NO，系统在 Redis 里记一个时间戳（`lead:refused:{user_id}`，TTL 1 小时）。接下来 3 分钟内不会再问同一个人，冷却期过了才重新尝试。这个窗口时长是通过 `ASK_COOLDOWN = 3` 控制的，单位是分钟。
>
> **这几层决策不是直接操作 Agent 的回复，而是通过 prompt 注入来影响 Agent**——如果决策为"适合留资"，就在系统指令里加一句"请在回答后自然引导留资"；不适合就加"专注解答用户疑问，先建立信任"。Agent 仍然是自由的，但有了这个上下文信号，回复会更贴合场景。"

### 可能的追问

**Q：冷却期过了又去问，用户再次拒绝怎么办？**
> 每次拒绝都会刷新 Redis 里的时间戳，冷却窗口重新开始计时。不会无限制问下去——如果用户持续拒绝，每次都是 3 分钟后才再问一次，不会在同一轮对话里反复骚扰。

**Q：那用户的意愿是通过什么来判断的？大模型？关键词匹配？**
> 还是走意图识别这条链路。用户说"改天再说"、"先不加微信"之类的，意图识别会返回 CONTACT_NO。跟普通 query 一样经过 intent × sentiment 矩阵，没有特殊处理。

**Q：怎么避免用户在还没了解产品的情况下就被引导留资？**
> 代码里 `_should_ask_contact` 的调用时机是在 Agent 回答**之前**，但实际上 `should_ask = True` 只是给 Agent 一个信号——"本轮适合引导"。Agent 的 system prompt 里有逻辑要求"先回答用户问题，再自然引导"，所以实际效果是先解答完疑问再要联系方式。另外 routing 上只有 CONSULT Agent（主力咨询）才会触发留资引导，GREETING 开场、ESCALATION 投诉处理都不会。

**Q：情绪是消极的情况下，有没有可能聊了几轮后情绪转好了再引导？**
> 目前每轮对话都会重新走完整的意图识别 + `_should_ask_contact` 决策链路。所以这一轮情绪消极不引导，下一轮用户情绪好了、意图也匹配了，就会自动变成引导。不需要额外的状态机来维护。

## 评测体系 + 数据闭环

### 面试话术

> **"这个项目里我觉得做得比较好的地方是搭了一套端到端的意图识别评测体系。** 因为意图识别的准确率很大程度上取决于 few-shot 示例设得准不准，线上出现 Bad Case 就要微调提示词。但怎么判断调整是变好了还是变坏了？不能靠感觉。所以我事先给每个意图都标注了常见的 query，做成 Golden Truth Set——一共 363 条人工标注用例。每次改完 prompt 就重跑这些测试集，看指标有没有回退。
>
> **指标上我重点看两个：Accuracy 和 Macro-F1。** 只看 Accuracy 是不够的——比如某类意图的测试用例特别多，Accuracy 可能被它拉上去，但小类全分错了也看不出来。所以我用 Macro-F1，先算每个类别自己的 F1（精确率和召回率的调和平均），再对所有类别取平均。这样每个类别是平等对待的，少数类的表现不好一眼就能看到。代码里是纯 Python 实现，逐类算 TP/FP/FN，然后算 precision、recall、f1。
>
> **除了意图评测，还搭了一套 LLM-as-Judge 的质量评估。** 用 LLM 当裁判，从相关性、准确性、完整性、有用性四个维度给 Agent 的回复打分（0 到 1）。为什么要用 LLM 判？人工标注太贵、太慢、还不一致，LLM judge 可以规模化地跑。
>
> **最重要的是基线对比和回归检测。** 每次评测结果会保存到 `data/eval/baseline.json`，下一次跑的时候自动跟基线比——哪个指标退了 5% 以上会标出来，还给出优化建议。这样改了提示词后是变好了还是变坏了，不是靠印象，而是有数据说话的。"

### 可能的追问

**Q：那你的 Golden Truth Set 只有 363 条，够覆盖所有场景吗？**
> 初期是够的，每个意图类别平均 30-40 条，能暴露大多数常见问题。但它不是一成不变的——线上 intent_logs.jsonl 会持续积累真实数据，定期从中抽 Bad Case 人工修正，补充到测试集里。所以测试集会随着时间越来越丰富。

**Q：为什么不用传统的分类模型（BERT）而是用大模型+评测？**
> 传统模型需要大量标注数据才能训练，项目初期没这个条件。大模型零样本就能用，但可靠性和可复现性确实不如小模型。所以我的思路是先靠大模型跑起来，同时通过评测体系把线上数据沉淀下来——intent_logs.jsonl 里的每条记录包含意图、情绪、推理过程。数据够了之后就可以用这些真实标注数据去微调一个本地的小模型，成本更低、响应更快、效果也更稳定。

**Q：LLM-as-Judge 本身也有偏差，怎么保证它的评分是可信的？**
> 你说得对，LLM Judge 本身确实有偏好。目前的做法是定期间隔抽一些评分结果跟人工复核对比，如果发现系统性偏差就调整 judge prompt。另外评测的核心不是追求评分绝对准确，而是**相对基准**——同一套 judge 逻辑，改 prompt 前后跑出来的分差是有参考意义的。只要 judge 设置不变，回归检测就能工作。

**Q：回归检测的阈值 5% 是怎么定的？**
> 经验值。设太小（比如 1%）会频繁报警，大部分是随机波动不是真退化；设太大又可能漏掉真正的问题。5% 在实际跑下来是一个比较合理的平衡点。代码里 `_detect_regressions` 就是干这件事的——比上一次或者基线，差超过 5% 就标记。

## 技术选型

### 面试话术

> **"整个项目的技术选型我遵循一个原则——在项目早期，用最轻量的方式验证价值，为后续演进留好接口。** 下面从模型、向量数据库、嵌入模型、检索方案几个维度来讲。"

---

### 1. 模型选型：为什么用 LLM，不用 BERT 这类分类模型？

> **"意图识别这块，最直觉的方案是 BERT/textCNN 做分类模型——轻量、快、稳定。但我没有选，原因有三：**
>
> **第一，冷启动问题。** 意图×情绪有 9×5=45 种组合，传统分类模型每个组合至少需要几十条标注数据才能训练，项目初期没这个条件。LLM 零样本能力就能跑起来，先上线验证。
>
> **第二，上下文理解能力。** 用户意图经常依赖多轮上下文——只说'那算了'，结合上几轮才能判断是放弃购买还是在压价。BERT 做单句分类天然丢失这个信息，LLM 可以用工作记忆里的多轮消息做判断。
>
> **第三，迭代速度。** 分类模型改一个类别要重新训练、重新部署。LLM 改几个 few-shot 示例就能调，配合评测体系验证效果。项目早期需求变化快，LLM 的灵活性很重要。
>
> **但这不是最终方案。** 线上 intent_logs.jsonl 一直在积累标注数据，够量之后可以蒸馏到一个小模型（比如 BERT 或者 0.5B 级别的 LLM）做本地部署，降成本、降延迟。"

### 2. 向量数据库选型：ChromaDB vs Milvus

> **"向量数据库我选了 ChromaDB，核心原因是轻量。** 项目早期没有专门的 DBA 资源，我不想为了一个向量检索就去维护 etcd + MinIO + Milvus 三套组件。
>
> ChromaDB 的优点是：嵌入式运行，不用单独起服务；数据存在本地 SQLite 里，备份和迁移就是一个文件；API 简单，get_or_create_collection 几行代码就能跑起来。对于知识库和情景记忆这个量级（几千到几万个片段），ChromaDB 的性能完全够用。
>
> **那 Milvus 在 docker-compose 里也部署了是怎么回事？** 那是给未来留的接口。当数据量上去之后——比如用户画像积累到百万级——ChromaDB 的 SQLite 底层扛不住高并发和复杂过滤，那时候切到 Milvus。Milvus 的优势在于分布式、向量+标量混合过滤、多向量检索。docker-compose.yml 里 etcd、MinIO、Milvus、Attu 都配好了，代码层面也预留了适配层，切换成本不大。"

### 3. 嵌入模型选型

> **"嵌入模型这块，我没有额外选第三方的 embedding API，用的是 ChromaDB 内置的 all-MiniLM-L6-v2。** 原因很简单——它够用、免费、不需要额外调接口。
>
> 项目早期做 RAG 召回，对 embedding 的要求是能大致区分语义相似度就行，不需要领域微调。all-MiniLM-L6-v2 是 Sentence-BERT 系列的轻量模型，384 维，在通用语义匹配上表现稳定。ChromaDB 内置在 query 时自动调用，对用户完全透明。
>

### 4. 为什么用混合检索，不用纯向量检索？

> **"纯向量检索的问题是：语义相似不等于精准匹配。** 比如用户问'退款流程'，向量检索可能召回了'退货政策'、'售后说明'这些语义接近的文档，但'退款'这个精确关键词反而权重不高——因为 embedding 是连续向量，不是精确匹配。
>
> **BM25 关键词检索正好补这个短板。** 它基于词频和逆文档频率，专有名词和精确术语的命中非常准。'401 错误'、'退款到账'这类场景，BM25 的召回精度远超向量检索。
>
> **所以我用混合检索——两者都跑，RRF 融合排序。** RRF（Reciprocal Rank Fusion）的好处是不需要归一化分数，只依赖排名。向量召回的文档和 BM25 召回的文档可能完全不重叠，RRF 给两者公平的融合机会。
>
> **上层再加 LLM 重排。** RRF 融合后取 Top-N，LLM 对这些候选按相关性二次打分排序。这样整个链路就是：多角度查询改写 → 向量+BM25 并行检索 → RRF 融合 → LLM 重排。每一层解决一个问题。"

### 5. BM25 关键词检索的具体实现

> **"BM25 索引是我纯 Python 手写的，没有用 Elasticsearch。** 原因很简单——为了一个关键词检索就引入 ES，太重了。项目里的知识库文档量级是几千到几万个 chunk，纯 Python 的 BM25 完全吃得消。
>
> **分词用的是 jieba。** 中文场景，jieba 是社区最成熟的方案。但我发现 jieba 对领域专有名词有误拆的问题——比如'退款政策'可能被拆成'退款'和'政策'，'401错误'被拆成'401'和'错误'。解决办法是在 BM25 初始化时注册了一个领域词典（`_DOMAIN_DICT`），把这些词手动加到 jieba 词表里。像'1-3个工作日'、'无理由退款'、'原路返回'这些客服高频词都提前注册了，保证分词精度。
>
> **如果量级上去了，后续可以切到 Elasticsearch。** ES 的 BM25 实现更成熟、支持分布式、还有 ik 分词插件。在 docker-compose.yml 里还没加 ES，但数据格式和接口设计已经为切换留了空间。"

### 6. RRF 的 K 值怎么设定

> **"RRF 的公式是 `score(d) = Σ 1 / (k + rank(d))`，K 控制了排名对分数的衰减速度。**
>
> K 越小，排名越靠前的文档分数权重越大，融合结果越偏向某一方的 top 结果；K 越大，排名靠后的文档也有机会进到最终结果里，融合结果更"民主"。
>
> 代码里 K 默认是 60，为什么是 60？没有什么玄学——这是 BM25 论文作者在 TREC 数据集上调参的经验值。后面也试过 30 和 100：K=30 时结果更偏向向量检索的 top 结果；K=100 时 BM25 的低排名文档更容易被带进来。60 是一个平衡点，在我们的场景下召回率和精度都 OK，就没有动它。
>
> 面试官如果问 K 值的调参思路，回答要点就是：**K 控制融合的"民主程度"，选 60 是因为论文经验值 + 实测平衡，线上数据积累多了可以用 Grid Search 找到最优值。**"

### 7. BM25 的 k1 和 b 怎么设定

> **"BM25 有两个核心参数：k1 和 b，代码里分别是 1.5 和 0.75——这是 BM25 论文里的经典默认值。**
>
> **k1（默认 1.5）** 控制词频饱和曲线。k1=0 时退化成二元模型（只看词有没有出现，不看出现几次），k1 越大，高频词的分数越不会被抑制。设 1.5 是一个中庸值——允许词频一定程度地累加分数，但不会让反复出现的"的、了"这类高频词主导结果。极端情况下如果文档里同一个关键词出现 10 次和出现 3 次，分数差距不大。
>
> **b（默认 0.75）** 控制文档长度归一化。b=0 时不考虑文档长度，b=1 时完全归一化。设 0.75 意味着：长文档有一定惩罚（因为越长越容易包含更多词），但不会完全抹平长度差异。如果一个 5000 字的文档和一段 50 字的 FAQ 都包含"退款"，b=0.75 会让短文档稍微占优，这符合检索场景——用户希望快速找到精确答案，不是长篇大论。
>
> **为什么没调？** 项目初期知识库文档类型比较统一（产品 FAQ、退款政策、使用指南），文档长度分布相对集中，默认值表现正常。如果后续要加入大量异构文档（比如用户手册长篇 vs 客服话术短篇），长度差异大了，b 值可能需要调到 0.8-0.9，让归一化更激进。这也是预留了参数接口，改个数值就行。

### 8. 对切分的chunk_size为什么是500

> **"chunk 大小 500 字符，overlap 50 字符。这是综合考虑 LLM 上下文窗口和检索粒度后的折中。**
>
> 500 字符大概相当于中文 200-300 个字，这个长度对于一个"知识点"刚刚好——比一句话长、比完整文档短。LLM 重排时能完整理解这块内容，不会因为太长而丢失重点。
>
> overlap 50 字符是为了防止切分把关键信息拦腰截断。比如"退款需要 3 个工作日到账"这句话如果正好被切分切成两半，下一块的开头补了上一块的尾巴 50 个字，就能保留上下文连续性。
>
> 切分策略是递归切割 + 贪心凑块的思路：先用段落符 `\n\n` 切，不行再降级到 `\n`、句号、感叹号、逗号，最后兜底按字符硬切，保证不会因为一个大段落无法切分而失败。然后贪心地相邻碎片凑到尽量接近 500 字符但不超出，最后加 overlap。这套逻辑在 `_chunk()` 方法里，没有用 LangChain 的切割器——因为不需要依赖一个几十 MB 的库来实现几十行能搞定的事情。"

### 9. 为什么不用 LangChain / LangGraph

> **"这是一个很好的问题。LangChain 和 LangGraph 确实是目前 AI 应用开发的主流框架，我没有用它们，原因有三：**
>
> **第一，抽象层次带来的调试成本。** LangChain 的抽象链（Chain、Runnable、Callback）封得比较厚，线上出了问题，你要从自己的业务代码一路追到 LangChain 的源码才能定位。比如 LLM 输出的解析失败、Callback 没触发、Memory 的 key 冲突——这些在框架里是黑盒。项目初期我们需要快速迭代、频繁调 prompt，每一层抽象都是排查问题的阻力。自己手写，每一行代码都知道在干什么，出问题能秒定位。
>
> **第二，80% 的场景用不到框架的复杂度。** 我的核心流程就四步：意图识别 → Agent 路由 → RAG 检索 → 留资决策。这些用 asyncio 就能串起来。LangChain 的 Chain、LangGraph 的 StateGraph、Agent Executor 确实可以编排更复杂的 DAG，但当前项目没有复杂的循环推理、多步工具调用、状态机切换的需求。用框架反而被它的抽象约束了。
>
> **第三，依赖管理。** LangChain 的依赖树很重——你装一个 langchain，它会拉 langchain-community、langchain-core、各种 integration 包。而且 LangChain 的 API 变动频繁，0.1 到 0.2 到 0.3 大版本之间 breaking change 不少，锁版本升级都是负担。我们的依赖就是 anthropic + redis + chromadb + jieba，精简直观。
>
> **但我不排斥框架。** 如果后续项目复杂度上去了——比如需要多步 Agent 循环、人机协作、复杂的 DAG 流程——LangGraph 是一个合理的选择。当前阶段，手写代码的灵活性和可控性大于框架带来的便利。"

### 10. 嵌入模型为什么不选 BGE（对中文语义理解更好）

> **"BGE 系列（BAAI General Embedding）确实是中文 embedding 的 SOTA，BGE-large-zh 在 C-MTEB 上的表现明显优于 all-MiniLM-L6-v2。没有用它，纯粹是 ChromaDB 的限制。**
>
> ChromaDB 内置的 embedding 模型是 all-MiniLM-L6-v2，API 层没有开放"更换 embedding 模型"的接口。你用 ChromaDB 的 `collection.add(documents=[...])`，它自动调内置模型做向量化，你没办法插一个 BGE 进去。除非你改成先外部算好向量再通过 `embeddings` 参数传进去。
>
> **所以这个问题的本质不是 all-MiniLM vs BGE 的取舍，而是 ChromaDB vs 其他向量数据库的取舍。** 既然选了 ChromaDB 的轻量嵌入式方案，那就接受它内置的 all-MiniLM。在通用英文场景和常见中文表达上，all-MiniLM 的表现是够用的——召回 Top-5 里至少能命中 3-4 条相关文档。
>
> **但 BGE 确实是我规划中的优化方向。** 目前有两个方案在考虑：
>
> 方案 A：**外部算 embedding**。用 FlagEmbedding 库加载 BGE-large-zh，离线算好所有 chunk 的向量，通过 ChromaDB 的 `embeddings` 参数传入。这样不换数据库，只换 embedding 模型。
>
> 方案 B：**切到 Milvus**。Milvus 原生支持 BGE、OpenAI、Cohere 等多种 embedding 模型，直接配置即可。docker-compose.yml 里的 Milvus 已经部署好了，切换成本主要是数据迁移。
>
> 两套方案都不需要改业务逻辑——向量检索的输入输出接口是一样的，换个 embedding 模型不影响上层代码。这就是分层设计的好处。目前没有切的原因和之前一样：**在 all-MiniLM 的召回效果出现明显瓶颈之前，不值得为了一个模型替换去动基础设施。**"

### 11. 关键词检索的具体实现

> **"关键词检索部分我关注两个核心点：分词和评分算法。**
>
> **分词用的是 jieba。** 中文场景下 jieba 是社区最成熟的方案，对中文的分词支持好，而且支持自定义词典。我注册了一个领域词典，把 '无理由退款'、'1-3个工作日'、'401错误' 这些客服高频词加到 jieba 词表里，防止被误拆成单个字或者错误切分。
>
> **评分算法用的是 BM25。** BM25 的核心思想是综合三个因素来打分：
>
> 第一个是**词频（TF）**——关键词在文档里出现得越多，分数越高。但不是线性关系，用 k1 参数控制饱和曲线，出现 3 次和出现 10 次的分数差距不会太大。
>
> 第二个是**稀缺度（IDF）**——关键词在整个文档集合里越稀缺，权重越高。像 '的'、'是' 这类停用词在所有文档里都出现，IDF 接近 0；像 '401错误' 这种只在少数文档里出现的词，IDF 就高，命中了对分数贡献很大。
>
> 第三个是**文本长度归一化**——用 b 参数控制。同样的关键词出现在短文档和长文档里，短文档的分数更高。因为短文档里出现关键词更能说明这篇文档就是讲这个的。
>
> 公式就是 `score = IDF × [freq × (k1+1)] / [freq + k1 × (1 - b + b × docLen / avgDocLen)]`。
>
> **关于倒排索引。** 坦白说，我当前没有维护倒排索引。每次搜索是遍历所有文档逐篇算 BM25 分数。对于几千个 chunk 的量级，遍历的开销可以接受——一次 BM25 检索大概几十毫秒。但如果数据量上到几十万篇，没有倒排索引性能就会劣化。后续优化方向有两个：一是把 token → doc_list 的倒排索引补上，二是如果量级再大就直接切到 Elasticsearch，它内置了成熟的倒排索引和 BM25 实现。
>
> **那为什么现在不用 ES？** 和数据量一样的原因——几千个 chunk 专门跑一个 ES 集群太重了。Elasticsearch 是分布式搜索，要维护集群、分片、索引生命周期，项目早期没必要引入这个复杂度。纯 Python 实现的 BM25 加上 jieba，代码不到 200 行，能解决问题，也能讲清楚原理，比上来就上 ES 更务实。"

### 12. 文档格式支持与表格处理

> **"目前支持四种文档格式：PDF、Word（.docx）、纯文本（.txt / .md）和 JSON。**
>
> - **PDF**：用 pdfplumber 提取文本，按页拆分，每页作为一篇独立文档。pdfplumber 的好处是既能提取正文文本，也能提取表格。
> - **Word（.docx）**：用 python-docx 解析，按文档的段落和表格保持原始顺序遍历，合并成一篇文档。
> - **纯文本 / Markdown（.txt / .md）**：整个文件作为一篇文档，文件名作为标题。
> - **JSON**：支持批量导入，格式是 `[{"title": "...", "content": "..."}]` 数组。
>
> **关于表格，我做了特殊处理。** PDF 和 Word 里的表格如果直接当纯文本提取，列和行的对应关系就丢了——'价格: 9800元, 库存: 50台' 这种信息，纯文本提取后可能变成无结构的字符串，语义信息严重丢失。
>
> 我的做法是 **把表格统一转成 Markdown 格式**。不管是 PDF 里的表格还是 Word 里的表格，提取出行列数据后，走同一个 `_table_to_markdown()` 函数转成 Markdown 表格语法：
>
> ```
> | 产品 | 价格 | 库存 |
> |------|------|------|
> | A    | 9800 | 50   |
> | B    | 12800| 30   |
> ```
>
> 这样做的考虑有两个：一是 Markdown 表格保留了二维结构，LLM 能理解行列对应关系；二是后续做 RAG 检索时，'9800' 和 '50' 这些数值能跟表头和上下文关联上，不会被当成孤立的数字。
>
> **但目前的实现还有不足**——表格转 Markdown 后再去做 chunk 切分，如果表格比较大，可能会被拆成两半，导致语义断裂。比如一个 10 行的价格表被切成 6+4 两段，第二段就丢了表头。目前能想到的优化方向是：对大表格单独处理，作为一个不可分割的 chunk 整体入库，不做切分。这块还没有实现，是已知的优化点。"

| 表格切分的问题？ | 大表格可能被 chunk 切分拦腰截断，后续优化方向是表格整体作为一个 chunk 不入切分 |

### 13. 文档的切割策略

> **"我的切割策略分三个阶段：优先按标点符号递归切 → 贪心凑 chunk → 加 overlap 保证语义完整。**
>
> **Phase 1：递归切分。** 不搞一刀切，而是按分隔符优先级逐级递归切割。优先级从粗到细：段落（`\n\n`）→ 行（`\n`）→ 句号（`。`）→ 感叹号（`！`）→ 问号（`？`）→ 英文句号（`.`）→ 英文感叹号（`!`）→ 问号（`?`）→ 空格 → 最后兜底按字符硬切。每级切完后看碎片大小，如果还超过 500 字符就降级到下一级继续切。这样切出来的碎片都是在"自然边界"上的，不会把一句话拦腰截断。
>
> **Phase 2：贪心凑 chunk。** 递归切出来的碎片往往很小，直接入库检索粒度太细了。遍历碎片列表，尽可能把相邻碎片合并到一起，让每个 chunk 尽量接近 500 字符但不超出。如果加一个碎片就会超，那就当前块结掉，开新块。这就叫贪心凑块——每次都尽量填满，不浪费容量。
>
> **Phase 3：加 overlap。** 相邻 chunk 之间重叠 50 字符。把前一个 chunk 的最后 50 个字拼到当前 chunk 的开头。这么做的目的是防止关键信息正好卡在两个 chunk 的边界上——比如"退款需要 [切分点] 3 个工作日到账"，不加 overlap 的话，搜"3 个工作日"可能就召回不到。加了 overlap，边界信息两边都有，召回率更高。
>
> **500 字符这个值**是基于场景的经验值。中文 500 字符大概 200-300 个字，对于一个"知识点"来说刚刚好——比一句话丰富、比完整文档聚焦。LLM 重排时这个长度能完整理解又不丢失重点。chunk 太小的话，检索引擎容易召回到碎片信息，缺乏上下文；chunk 太大，一个 chunk 包含多个主题，检索又容易误匹配。
>
> **这套切分代码全部自己手写，在 _chunk() 方法里大概 80 行。** 没有用 LangChain 的 TextSplitter——递归切割 + 贪心合并 + overlap 这个需求很直接，自己写比引入一个几十 MB 的依赖更可控。"

### 14. 文件上传的限制

> **"项目不做大文件上传，不支持秒传、断点续传这些功能。当前在代码里写死了最大 10MB 的限制——上传接口收到文件后先判断字节数，超过 10MB 直接返回 413 状态码。**
>
> **为什么不做？** 这个项目的核心场景不是文件管理，而是智能客服的检索增强生成。关注的重点是文档内容怎么切分、怎么检索、怎么帮大模型减少幻觉。文件上传只是一个数据入口，够用就行。
>
> **10MB 够用吗？** 对于客服知识库的场景——产品说明、FAQ、退款政策——单篇文档很少超过 10MB。如果真有超大文档（比如几百页的用户手册），建议先手动拆成多个小文件再上传。后续如果需要支持大文件，可以考虑分片上传 + 后端合并，但目前没有这个需求优先级。"

### 15. 文档更新策略

> **"文档更新的场景我考虑过，方案很简单——删除旧文档，重新上传即可。**
>
> 导入接口（`POST /knowledge/upload`）返回每篇文档的 `doc_id`，这是一个基于文档标题和内容前 200 字算的 MD5 哈希值。删除接口（`DELETE /knowledge/{doc_id}`）接收这个 doc_id，从 ChromaDB 和 BM25 索引中同步删除该文档的所有 chunk。
>
> 所以更新流程就是两步：先调 DELETE 删旧版，再调 POST 传新版。ChromaDB 和 BM25 索引会同步清理干净，不会留下脏数据。
>
> 那为什么不做增量更新或者版本管理？因为客服知识库的文档是人工维护的——比如运营更新了退款政策，改完后重新上传一次。文档更新的频率很低（周级甚至月级），不是用户实时生成的内容。为这种低频场景做一个复杂的版本管理系统，投入产出比不高。简单的删+传，逻辑清晰、实现可靠、不会出错。"

### 16. LLM 调用失败的降级策略

> **"我们的降级方案是用本地 Ollama 做远程大模型的兜底。** 当前项目的主力模型是远程 API（Claude），但完全依赖远程服务是有风险的——网络波动、API 限流、服务宕机都可能导致线上对话中断。所以我在本地部署了 Ollama 服务，作为降级链路。
>
> **降级链路分两层：**
>
> **第一层：意图识别降级。** IntentRecognizer 的 LLM 调用包在 try-except 里。如果远程 API 抛异常或者超时，请求自动路由到本地 Ollama（qwen2.5-7b）。Ollama 返回的结果格式跟远程保持一致——intent + sentiment 两个标签。这个降级对上游是透明的，Agent 层感知不到切换。如果 Ollama 也失败了，才返回安全默认值（CHITCHAT + NEUTRAL）。
>
> **第二层：Agent 执行降级。** Agent 调用远程 API 失败时，同样切到 Ollama 生成回复。如果一个 Agent 类型（比如 EscalationAgent）彻底挂了，还有第三道防线——`_execute()` 里的 Agent 级别降级，自动降级到 ConsultAgent 处理，而不是直接给用户报错。
>
> **Ollama 部署在 docker-compose 里，跟主服务同机。** 模型用的是 qwen2.5-7b-Q4_K_M 量化版，显存占用大概 6GB，响应延迟 2-3 秒。这个模型在通用对话和意图识别上的能力足够做降级兜底，不会出现答非所问的情况。切换到 Ollama 时会在日志里标记，方便监控和告警。
>
> **一句话总结：远程 API 正常时 Ollama 闲置待命；远程挂了自动切 Ollama，挂了也不中断对话；两个都挂了才返回兜底话术。**

### 17. 并发场景（薄弱环节 —— 诚实面对 + 改进思路）

> **"这块确实是我当前项目考虑得比较少的地方。我直接说现状，以及我知道应该怎么改。"**

**Q1：多用户同时触发工作记忆压缩，Redis 操作有竞态条件吗？**

> **"有，现在是有问题的。** `_compress()` 里是先 llen 判断长度，再 lpush 写入，再 delete + lpush 重置——这三步不是原子操作。如果两个请求同时 llen 发现超过阈值，会触发两次压缩，后一次压缩可能把前一次的操作覆盖掉。
>
> 但线上实际影响不大，因为压缩是偶发的（15 条才触发一次），而且即使漏掉一次压缩也不影响核心功能，最多是多消耗一点 token。但如果要修，方案是用 Redis Lua 脚本把 llen 判断 + 压缩操作包成一个原子操作，或者加一个分布式锁（SETNX），保证同一会话同一时刻只有一个压缩在执行。"

**Q2：update_profile 是异步的，并发调用会不会覆盖数据？**

> **"会。** `update_profile()` 是先 delete 再 add——两个调用之间如果有另一个请求也调 update_profile，后一个可能把前一个的画像数据删掉了还没写回去。同样的，加锁可以解决。但画像覆盖更新的频率不高（每次对话才更新一次），实际碰到的概率很低。"

**Q3：ChromaDB 的 SQLite 后端能支持高并发吗？**

> **"不能。** ChromaDB 本地模式下底层是 SQLite，SQLite 的写入是串行锁的，并发写超过几十 QPS 就可能报 database is locked。这也是为什么我在 docker-compose 里部署了 Milvus——如果线上并发量真上去了，直接切到 Milvus，它支持分布式写入。当前单机演示和低并发场景，ChromaDB 够用。"

**Q4：FastAPI 是异步的，但你的 Redis 操作用的是同步客户端，会阻塞事件循环吗？**

> **"会。** 代码里用的是 `redis.Redis`（同步客户端），在 async 函数里直接调用会阻塞事件循环。影响不大因为 Redis 操作本身很快（微秒级），但如果同时有几十个请求进来，事件循环被 Redis 调用卡住就会有明显延迟。正确的做法是用 `redis.asyncio.Redis`，整个替换成本不高——API 基本一致，改一下 import 和 await 就行。"

**Q5：LLM API 有速率限制，多个用户同时请求怎么办？**

> **"当前没有做限流和排队，是直接并发的。** 如果用户量大了，远程 API 会返回 429 限流错误。解决方案是加一个请求队列——把 LLM 请求放到一个带并发上限的 asyncio.Queue 里，控制同时进行的 LLM 调用数（比如 5 个），超出的请求排队等待。配合 Ollama 降级，排队超时的请求切成本地模型，保证不丢请求。"

**Q6：整体架构是单进程部署，挂了怎么办？**

> **"当前确实是单点。** 没有多副本、没有负载均衡、没有健康检查。如果进程挂了整个服务就不可用。如果要上生产，标准做法是：前面加一层 Nginx（nginx.conf 已经有了）做反向代理和限流，后面起多个 uvicorn worker 或多个容器实例，进程管理用 Docker 的 restart policy（docker-compose 里已经配了 restart: unless-stopped）。这一步改动不大，基础设施基本就绪。"

### 18. 文档重复上传

**Q：有没有考虑过用户重复上传文档的情况？**

> **"考虑过。我的做法是全文 MD5 作为文档 ID，上传时先查 ChromaDB 里这个 doc_id 是否已存在，存在就直接跳过。API 返回值里会带一个 `skipped` 字段，调用方能看到跳过了多少篇重复。这样同一篇文档重复上传不会产生脏数据。"**
