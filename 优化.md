# EchoMind 优化计划

> 基于项目现有代码的三个优化方向

---

## 一、迁移至 Milvus 统一检索平台

**当前状态**：ChromaDB（向量检索） + rank_bm25（关键词索引，进程内存）

**核心问题**：
- 双存储架构下文档删除无法同步——ChromaDB 按 MD5 删除向量很方便，但 rank_bm25 的索引是进程内全局结构，无法单文档删除，需维护额外的 deleted_set 并定期全量重建
- 多了一个需要维护一致性的组件，且 rank_bm25 索引随系统重启丢失

**优化目标**：统一迁移至 Milvus 作为知识库底座，利用其 Dense + Sparse 双向量能力实现混合检索，内置 BM25 关键词检索支持原生单文档删除

### 嵌入模型

统一使用 `BAAI/bge-base-zh-v1.5`（中文优化，768 维）：

```python
from sentence_transformers import SentenceTransformer

model = SentenceTransformer("BAAI/bge-base-zh-v1.5", device="cpu")
embeddings = model.encode(chunks).tolist()
```

**选型理由**：
- 在 C-MTEB 中文 benchmark 上综合表现领先
- 768 维比原本 all-MiniLM-L6-v2 的 384 维语义表示能力大幅提升

### Milvus Collection 设计

```
Collection: knowledge_base

字段：
  chunk_id      VARCHAR(64)             ← MD5(title + chunk_index + chunk[:50])
  chunk_text    VARCHAR(2000)           ← 文档片段原文
  title         VARCHAR(256)            ← 文档标题
  file_hash     VARCHAR(64)             ← 文件 MD5，用于精准删除
  chunk_index   INT64                   ← 片段序号
  total_chunks  INT64                   ← 总片段数
  dense_vector  FLOAT_VECTOR[768]       ← BGE-base-zh 语义向量
  sparse_vector SPARSE_FLOAT_VECTOR     ← 内置 BM25 关键词向量（Milvus 原生）

索引：
  dense_vector  → IVF_FLAT / HNSW       ← 语义检索
  sparse_vector → 内置反向索引          ← 关键词检索

删除文档（一行搞定）：
  collection.delete(expr='file_hash == "abc123"')
  → 向量 + BM25 索引同时清除，无一致性问题

检索（Hybrid Search + RRF）：
  hybrid_search(
    reqs=[dense_req, sparse_req],       // 双路并行
    rerank=RRFRanker(),                 // Milvus 内部 RRF
    limit=top_k
  )
```

### 分词器配置

Milvus 支持自定义分词器，在创建 Collection 时通过 `analyzer_params` 配置：

```python
schema = CollectionSchema([
    # ...
    VarCharField(name="chunk_text", max_length=2000,
                 enable_analyzer=True,
                 analyzer_params={
                     "type": "chinese",     # 使用 jieba 分词
                 }),
])
```

可维护自定义词典（`data/dict/custom_words.txt`），将产品型号（`EP-301`）、错误码（`ERR-2451`）、业务专有词等加入，防止被错误切碎。

### 多格式文档支持

| 格式 | 处理方式 |
|------|---------|
| `.txt` / `.md` | UTF-8 解码，全文作为一篇文档 |
| `.json` | 解析为 `[{title, content}, ...]` |
| `.pdf` | pdfplumber 逐页提取文本，**识别表格区域转为 Markdown 表格** |
| `.docx` | python-docx 提取段落文本 + 表格（转 Markdown 表格） |

PDF 中的表格处理策略：
- pdfplumber 能识别页面中的表格边界，提取出行列数据
- 行列数据组装为 Markdown 表格文本存入 chunk，保证检索时能匹配到行列内容

### 文件删除机制

```
上传流程：
  读取文件内容 → 计算 MD5 (hashlib.md5) → jieba 分词 → 现有切片逻辑
  → BGE 编码 → Milvus upsert（metadata 含 file_hash）

删除流程：
  POST /knowledge/delete {"file_hash": "abc123"}
  → collection.delete(expr='file_hash == "abc123"')
  → 这个 chunk 对应的稠密向量、稀疏向量、metadata 全部清除
```

### 架构变化

```
优化前：
  ChromaDB（知识库向量） ──┐
                           ├── 手动 RRF 融合
  rank_bm25（关键词索引）──┘        ↑ 忘记重建就一致性问题

优化后：
  Milvus: knowledge_base（dense + sparse 一体）
    → hybrid_search + RRF（Milvus 内部保证一致性）
    → delete(expr=...)（向量 + 关键词索引同时删除）
```

## 二、文档切割 + 父子分块

**当前状态**：按句号切分，每片 500 字，无 overlap，无分层

**优化目标**：递归分割优先保留语义边界，token 兜底保证不丢内容，overlap 防止上下文断裂；子块命中多个同源父块时自动合并

**改动点**：

| 文件 | 改动 |
|------|------|
| `mcp/knowledge_base.py` | 重写 `_chunk_text()`，改为递归分割（段落 → 句子 → token）；引入 overlap 参数；写入时保留 parent-child 关联关系 |

**存储结构**（同一 collection，metadata 区分）：

| id | type | content | metadata |
|----|------|---------|----------|
| `parent_001` | parent | 退款政策全文 1200 字 | `{type: "parent", doc_id: "001"}` |
| `child_001_0` | child | "用户在购买后7天内..." 500 字 | `{type: "child", parent_id: "parent_001", doc_id: "001"}` |
| `child_001_1` | child | "退款申请提交后..." 500 字 | `{type: "child", parent_id: "parent_001", doc_id: "001"}` |

**合并逻辑**：

```
用户 query → 向量检索
  ↓
召回子块 [child_001_0, child_001_1]
  ↓
两个子块的 parent_id 都是 parent_001 → 触发合并
  ↓
去 ChromaDB 查 parent_001 全文 → 返回父块
```

## 三、评测体系：检索层评估 + 全链路可观测

**当前状态**：只有 LLM-as-Judge 生成层评估（四维评分），没有检索层评估；没有 trace 可观测

**优化目标**：构建检索层评测集，计算 Recall@K；集成 Langfuse 做全链路 trace，定位召回瓶颈

### 改动点

| 文件 | 改动 |
|------|------|
| `requirements.txt` | 加 `langfuse` |
| `mcp/tool_manager.py` | `search_with_rewrite()` 加 `@observe` 记录子查询、召回数、各阶段耗时；检索完成后异步调 LLM 做相关度评分 |
| `evaluation/evaluator.py` | 新增 `RetrievalEvaluator` 类：加载评测集（Query + 知识来源），跑批量召回算 Recall@K；`judge()` 结果写回 Langfuse score |
| `.env` | 加 `LANGFUSE_PUBLIC_KEY` / `LANGFUSE_SECRET_KEY` |

### 评测集结构

```json
[
  {
    "query": "退款多久到账",
    "source_doc": "退款政策"
  },
  {
    "query": "登录报错401怎么解决",
    "source_doc": "技术故障排查"
  }
]
```

不绑 chunk id，只绑知识来源。chunk 切分策略变了也不影响评测集复用。

### 检索层指标：Recall@K

```
Query "退款多久到账" 的知识来源是 "退款政策"

召回 Top-5 结果：
  1. 退款政策（chunk_0）✅
  2. 会员积分          ❌
  3. 退款政策（chunk_1）✅
  4. 配送说明          ❌
  5. 账户安全          ❌

Recall@5 = 知识来源被覆盖的次数 / 总知识来源数 = 1/1 = 100%

对比实验：
  纯向量召回:          Recall@5 = 60%
  + BM25:             Recall@5 = 80%
  + RRF:              Recall@5 = 85%
  + 查询改写:           Recall@5 = 90%
```

### Langfuse trace 结构

```
rag_search
  ├── rewrite_query → 子查询数组
  ├── vector_search → 召回 N 条
  ├── bm25_search   → 召回 M 条
  ├── rrf_merge     → 候选集 K 条
  ├── parent_merge  → 合并后 P 条
  └── llm_rerank    → 最终 Top-K 条
```

某条 Query 最终 Recall@K 偏低时，通过 trace 可定位是**检索阶段没召回到**还是**重排阶段排掉了**。
